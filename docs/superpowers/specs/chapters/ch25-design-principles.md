# ch25 Spec — 设计原则横向收束：三条设计线 + A–G 矩阵 + 术语表

## 目标
全书最后一章,横向收束:把 24 章串成三条贯穿的设计线 + 一张 A–G 约束×章节矩阵 + 一份术语表。收束章无新源码 codefile(引用全书),重点是**归纳准确**——每条主线的章节序列、每个约束的治理章节,都要与前面章节实际内容自洽。

## 结构（收束章,非标准 5 段）
lead(全书收成几条线) → analogy(星座连线) → macro(一句话总纲) → 三条主线(各 h2 + vflow 串章节) → A–G 约束矩阵(card collab 风格,7 约束) → design(元设计哲学 + A–G 全 7 badge) → key(术语表)。

## 三条主线（章节序列,须核对准确）
1. **缓存神圣线**:ch6(system prompt 逐字节稳定) → ch15(压缩唯一例外) → ch18(控制命令旁路不进历史) → ch21(cron 独立会话不镜像) → ch23(技能注入 user message)。
2. **自我进化线**:ch9(学习 nudge 写技能) → ch10(Curator 园丁) → ch11(记忆 MEMORY/USER) → ch12(跨会话搜索 FTS5)。
3. **窄腰线**:ch4(窄腰哲学) → ch8(工具系统 + Footprint Ladder) → ch16(终端多后端边缘子类) → ch17(28+ 适配器统一 MessageEvent) → ch23(插件/技能/MCP)。

## A–G 约束矩阵（须核对每个约束的治理章节准确）
- A 中间遗失(引入 ch2)→ 治:ch4/8/23 窄腰精简核心工具、ch15 压缩、ch13 委派关中间过程进子上下文。
- B 无状态(引入 ch2)→ 治:ch6 system prompt 身份、ch11 记忆、ch20 profile 状态盘、ch16 会话 spawn-per-call 快照。
- C 幻觉(引入 ch3)→ 治:ch8 工具拿真实数据、ch14 生成-验证分离。
- D 指令=数据(引入 ch3)→ 治:ch18 网关守卫、ch24 纵深防御。
- E 结构化输出脆弱(引入 ch3)→ 治:ch7/8 工具 schema + 在地修复。

注:ch14 = F·误差累积 + C·幻觉(以章节实际为准;若 design.md §5 写 ch14=B+D 则 §5 旧,以此矩阵为准)。

## 元设计哲学（三句记忆）
① 状态外置(内核无记忆,记忆/技能/profile 外部文件,模型是纯函数);② 上下文神圣(缓存前缀逐字节稳定,压缩唯一例外);③ 窄腰厚边(核心薄,能力在插件/技能/MCP 边缘)。安全横切:绝不让概率模型当裁判,信任边界钉在确定性代码。
