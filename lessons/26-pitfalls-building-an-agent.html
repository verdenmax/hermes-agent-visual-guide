自己做 agent 的避坑指南Pitfalls of building an agent

你做 agent 会踩的坑，几乎都是 LLM 七缺陷（A–G）在实践中的案发现场。这一章是 agent 工程的事故档案：把别人摔过的坑写成规章。

🔌 类比 · 飞行事故档案 / 黑匣子
你坐的每一班飞机都极其安全，但这份安全不是凭空来的——民航那本厚厚的规章，几乎每一条背后都压着一次真实的空难：有人摔了，调查员翻遍黑匣子，把血的教训写成一行「今后必须……」。规章从来不是工程师拍脑袋定下的禁令，而是用事故倒推出来的。

这一章就是 agent 工程的事故档案：后面每一条「坑」都不是抽象告诫，而是别人已经摔进去过的现场——缓存被击穿、自主循环跑飞、注入得手、JSON 解析崩掉。我们要做的，是像调查员读黑匣子那样，把每个坑还原到它撞的那条 LLM 约束上，再写成你能照做的规章。之所以用「事故档案」打比方，是想说清一件事：这些坑都已经有人替你踩过、代价也付过了，你要做的只是别再付第二遍。

🌍 宏观 · 坑的两个源头
把后面所有的坑摊开看，它们不是随机散落的一堆 bug，而是从两个源头里长出来的——

① A–G 七缺陷在实践中现身。模型本就无状态、注意力两头重中间轻、分不清指令与数据、按概率续写（第 2–3 章的 A–G）：单看一次调用就已脆弱，把它串成自主体后更脆弱。后面每一个坑，本质都是这七条里的某一条在你的代码里显了形。

② 三条设计线（缓存神圣 / 自我进化 / 窄腰）的反面。这三条主线（第 25 章）讲的是「该怎么做」，而坑往往就是逆着它们走：会话中途改前缀 → 破缓存神圣；把学到的东西塞进系统提示而不外置成文件 → 破自我进化；什么能力都往核心里塞 → 破窄腰。

所以避坑不靠死背一长串禁令，而靠一张能把坑映射回约束的地图——下面这张「坑地图」就是全章的总览。

A 类 · 上下文与缓存

第一类坑全部压在缓存神圣这条线上。它们的剧本如出一辙：你出于「让上下文更新鲜、更整齐」的好意去动了前缀或消息结构，却撞碎了「每个会话的 prompt 缓存逐字节不可侵犯」这条铁律——代价是成本翻倍、模型失忆、甚至偶发空响应。之所以把它排在第一类，是因为它最高频、也最贵：一次踩中，受伤的不是某一轮，而是整条会话往后的每一轮。四张卡分别对应缓存的四种破法——塞进前缀（A1）、中途改前缀（A2）、破坏消息结构（A3）、压缩抖动（A4），但根子都是第 25 章那条「缓存神圣」设计线的反面。先用四张陷阱卡速览，再逐个拆 ❌ 反例 → ✅ 正例。

A1把记忆 / 可变状态塞进 system prompt
症状每轮 token 成本不降反升、agent 记不住新学的东西。
根因B·无状态的错误解法——前缀每轮变、缓存永不命中，且改 prompt 等于无法演化。
对策状态外置到文件，system prompt 只放逐字节稳定的身份。
→ 章ch6 / ch11
A2会话中途改前缀（重排工具 / 插系统通知 / 刷新记忆）
症状长对话越聊越贵、某轮起成本突然翻倍。
根因会话中途改了前缀，缓存从那点起全废（撞缓存神圣线）。
对策绝不中途动前缀，只追加；要改只能在压缩边界。
→ 章ch6 / ch15
A3消息序列两个同角色相邻 / 中途注入合成 user
症状模型偶发返回空、然后莫名重试。
根因两个同角色相邻 / 中途注入合成 user message → provider 静默返回空、触发 empty-retry（E·结构化脆弱 + 协议要求）。
对策发送前跑一遍角色交替修复（删孤儿 tool、合并连续 user）。
→ 章ch7
A4压缩阈值设太低 / 每轮都压
症状压缩频繁触发、缓存反复失效、体感变卡变贵。
根因阈值太低 / 每轮都压 → 抖动，每次压缩都废一次缓存。
对策阈值默认 ~50%、连续 2 次无效压缩才停。
→ 章ch15

🔬 A1 · 记忆不进 system prompt，走 append-only 的对话流

最自然的「记忆」写法，是每轮开头把「用户叫 X、偏好 Y、上次聊到 Z」连同当前时间戳拼进 system prompt——直觉上越新鲜越好。可 system prompt 恰恰是缓存前缀的最前一段：它每轮变一个字节，整条前缀就每轮 miss，于是「记忆越多 → 前缀越长 → 每轮全价重算越贵」；更糟的是你想「改身份」也得动这一段、再砸一次缓存——这正是 B·无状态 最典型的错误解法。

Hermes 反着来：system prompt 按稳定度切成 stable / context / volatile 三段，会话开始时整段构建一次、缓存在 agent._cached_system_prompt 上，之后 mid-session 绝不重渲染。记忆 / 画像确实在 volatile 段（由 tools/memory_tool.py:567 的 format_for_system_prompt() 在 agent/system_prompt.py:426-433 注入），但取的是会话开始那一刻的快照、整会话不再变；新学到的东西不写回 system prompt，而是通过工具调用进入对话流（append-only），只往尾部追加、不动前缀。

❌ 反例 · 你可能会这么写

    )

🔬 A2 · 只在 deepcopy 副本上打断点，绝不改原列表

「顺手维护一下上下文」是 A2 的高发动作：插一条系统通知、按当前任务重排工具定义、把刚写的记忆刷新回历史中段——它们都改动了已被缓存的前缀。缓存是前缀缓存：从第一个变动的字节起，后面所有 token 的缓存断点全部作废、按全价重算，对话越靠后、被牵连重算的 token 越多，这一下可能贵到原来的好几倍。

Hermes 把这条纪律钉进代码：apply_anthropic_cache_control 第一步就 copy.deepcopy(api_messages)，所有 cache_control 断点只打在副本上，原始消息列表分毫不动——于是「打断点」这个动作本身永远不会污染被缓存的那份消息序列。真要重建，也只在压缩边界（第 15 章）一次性进行。

❌ 反例 · 你可能会这么写
# 反例：会话进行到一半，往历史中段插一条系统通知
messages.insert(3, {"role": "system", "content": "[配额仅剩 10%]"})
# 或：按当前任务重排 / 增删工具定义，顺手「整理」上下文

# 从插入 / 改动点起，后面每个 token 的缓存断点全部作废 → 全价重算

🔬 A3 · 发送前强制角色交替

provider（OpenAI / OpenRouter / Anthropic）都要求严格的角色交替：system 之后，user / tool 与 assistant 必须一来一回，不能两个 user 相邻、也不能出现没跟在「带 tool_calls 的 assistant」后面的 tool 结果。一旦违反，多数 provider 不报错而是静默返回空，再触发一轮 empty-retry——表现就是「偶发空响应 + 莫名重试」，这是 E·结构化脆弱叠加协议硬要求的典型现场。

Hermes 在每次发请求前用 repair_message_sequence 兜底：删掉孤儿 tool 消息、合并连续的 user，把实时历史强行捋成合法交替序列（带游标的变体 repair_message_sequence_with_cursor 在 :448，供流式续写场景使用）。它跑在 API 调用的最后一刻，专门兜住网关多队列重放、会话恢复、cron、宿主直接传入的已损坏历史。

messages = [

B 类 · 自主循环

第一类坑压在「缓存神圣」这条线上；第二类坑则发生在你把单次调用串成自主回路的那一刻。单次调用本身就有 C·幻觉、F·误差累积、G·运维难三个毛病（第 2–3 章），一旦让模型自己决定下一步、循环往复，这些毛病会被回路放大、复利：没有上限它就无限烧（B1），自报「已完成」你就信它（B2），让子代理共享你的上下文它就污染你的缓存（B3），把一条回路拉太长早期小错就一路放大（B4）。四张卡的对策其实是同一套自主性纪律——给回路钉死上限、把子代理隔离、只认可验证的把手。先用四张陷阱卡速览，再拆 B1 / B2 两个 ❌ 反例 → ✅ 正例。

B1自主循环没有硬上限
症状一个任务跑了几十上百轮停不下来、账单爆炸。
根因循环没有硬上限（G·运维 + F·误差累积）——模型自言自语、无限烧。
对策max_iterations + iteration_budget + 每轮中断检查给循环钉死上限。
→ 章ch5
B2把子代理的自然语言自报当成事实
症状子代理报告「已完成 / 已修复」，实际没做或做错，错误被当事实继续用。
根因信任子代理的自然语言自报（C·幻觉）——SELF-REPORTS 不可信。
对策要求可验证把手（URL / ID / 路径 / HTTP 状态），父代理独立核验。
→ 章ch14
B3让子代理共享父代理的上下文
症状父代理上下文被子任务的中间噪声淹没、缓存被污染、注意力下降。
根因子代理共享父上下文、不隔离（A·中间遗失 + 缓存线）。
对策子代理独立 context + terminal，只回摘要。
→ 章ch13
B4单条自主回路拉得太长
症状自主链路越跑越歪、后期全是基于早期小错的放大。
根因单条回路拉太长、误差累积无截断（F·误差累积）。
对策用委派把大任务拆成各自隔离、独立收敛的短回路。
→ 章ch13

🔬 B1 · 给自主循环钉死上限：迭代数 + 预算 + 每轮中断检查

写 agent 主循环最自然的姿势就是 while True:——不停地把消息喂给模型、执行它要的工具、再喂回去。可模型是按概率续写的（G/F）：一旦没有硬天花板，一个困惑的模型能自言自语几十上百轮，每一轮都是一次完整 API 调用，账单就这么爆了。这正是单次调用的毛病在自主回路里被复利放大的样子。

Hermes 把上限钉在核心循环的最顶端：那行 while 是三道相互独立的闸门同时成立才继续——api_call_count < max_iterations（硬迭代上限，默认 90，见 agent/agent_init.py:165）、iteration_budget.remaining > 0（与子代理共享的独立预算），以及进入循环体后每轮第一件事就是查 _interrupt_requested（用户一发新消息就能立刻打断）。末尾的 _budget_grace_call 只留一轮「收尾恩典」，让 agent 把最后一句话说完而不是半句猝死。三道闸门任一不满足，循环立即结束——再也不会失控烧钱。

# 反例：核心循环没有任何硬上限

    messages.append(resp)
# 没有 max_iterations / iteration_budget / 中断检查 → 几十上百轮停不下来、账单爆炸

🔬 B2 · 不信子代理的自报，只信可验证的把手

子代理干完活，会用自然语言给你一句总结：「文件已上传」「测试已修复」。最危险的写法就是按这句话的字面意思往下走——if summary.endswith("done"): mark_complete()。但子代理同样是 LLM、同样会幻觉（C）：它可能真心以为上传成功了，实际请求早已失败；它的 summary 是自报，不是已核验的事实。你把自报当事实，错误就从这里一路下传，最后 publish 给用户的是个假「成功」。

Hermes 的 delegate 工具在给子代理的 prompt 里把这条纪律写成一段明文：子代理的 summary 是 SELF-REPORTS、不是已核验事实；凡是有外部副作用的操作（HTTP POST/PUT、远程写、共享路径建文件、发布），都必须让子代理返回一个可验证的把手（URL / ID / 绝对路径 / HTTP 状态），父代理自己去核验——fetch 那个 URL、stat 那个文件、读回内容——确认无误之后才告诉用户成功。一句话：把手能被独立验证，自报不能。

result = run_subagent("把季度报告上传到 S3")

剩下两类坑（B3 / B4）共享同一条出路——委派即隔离。第 13 章的 delegate_task 给每个子代理独立的 context 与 terminal 会话，默认父代理只等子代理回一条摘要再继续：这同时治了 B3（子代理的中间噪声不会涌进父代理的缓存前缀）和 B4（把一条会无限放大误差的超长回路，拆成若干各自隔离、独立收敛的短回路）。两张图分别对应这两种破法。

B3 · 隔离即保护：子代理共享父上下文，中间噪声会涌进父缓存、污染前缀、拉低注意力；独立 context + terminal 沙箱让噪声留在子代理内部，父代理只收一条摘要、缓存保温。
B4 · 短回路：一条超长回路里早期小错会被后续每一步放大，越跑越偏；用委派把大任务拆成各自隔离、独立收敛的短回路，误差没有跨回路累积的机会。

🔬 C1 · 别什么能力都加成核心工具，按 Footprint Ladder 往下沉

给 agent 加能力最顺手的写法，就是 registry.register(name=..., toolset=...) 再开一个核心工具——需要什么就加什么。但 Hermes 的核心是一条窄腰，AGENTS.md 说得很直白：「every model tool we add is sent on every API call, so the bar for a new core tool is high」。每个核心工具的 schema 都随每一次 API 调用发给模型，等于向每一个用户、每一轮对话收一笔「schema 税」；工具越堆越多，模型还会在一堆选项里选错、注意力被稀释——这正是 A·中间遗失 在工具层的现身。

🔬 C3 · 给工具一个统一契约：handler 全返回 JSON string，dispatch 统一包裹异常

工具多了之后，最容易出事的不是某个工具本身，而是它们返回的形状各不相同：有的 return {"ok": True}（dict）、有的 return "ok"（str）、有的干脆忘了 return（None）。dispatch 想把结果塞回对话流，就只能到处 isinstance 特判，漏一个分支就偶发崩溃——这是 E·结构化脆弱 的典型现场。

🔬 A1 · Keep memory out of the system prompt; route it through the append-only stream

The fix isn't "tell the model not to trust it" (it can't tell), it's to pin the trust boundary in a deterministic code gate: a dangerous action, whatever token it came from, still passes detect_hardline_command / detect_dangerous_command (approval.py:333 / :657), a gate independent of "who said it"; layered with injection isolation (ch18: the gateway uses resolve_command to parse control commands explicitly and route them out-of-band, out of the data stream). In one line — don't let the model be the judge.
