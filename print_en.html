What is Hermes

Hermes is Nous Research's self-improving personal AI agent. Its tagline calls it “the only agent with a built-in learning loop”: while you use it, it creates skills, improves them, nudges itself to persist knowledge, searches its own past conversations, and grows a deepening model of you across sessions. One agent core runs the CLI, the messaging gateway, the TUI and a desktop app.

🔌 Analogy
A normal agent is a temp worker with daily amnesia: what you taught it today must be re-taught tomorrow. Hermes is more like a seasoned employee who keeps notes: after each tricky job it writes “how to solve this class of problem” into a skill book and “facts about you” into memory. Next time, it flips to the notes and finishes in a few steps — better the more you use it.

What it actually is

In one line: an agent that turns experience into capability. It doesn't train models for you; it turns proven approaches into reusable procedural knowledge as you work. This matters because LLMs have a hard limitation — they remember nothing between two calls. Every Hermes “evolution” mechanism is, at heart, a fight against that amnesia.

🌍 Big picture
One AIAgent core is shared by the CLI, the gateway (Telegram/Discord/Slack…), the TUI and a desktop app. Capability mostly lives at the edges — via skills, plugins and MCP — while the core stays narrow. Plus one iron rule: per-conversation prompt caching is sacred, and almost every design bends around it.

Why it isn't tied to your laptop

Many agents only run on the one machine you keep open. A core goal of Hermes is to free the agent from a single box: it runs on a $5 VPS, a GPU cluster, or serverless infra that hibernates to near-zero cost when idle and wakes on demand. So you can message it from Telegram while it works on a cloud VM — close your phone and it keeps going.

It's also model-agnostic: Nous Portal, OpenRouter (200+ models), OpenAI, your own endpoint — switch with one hermes model command, no code changes. This “freedom of place + freedom of model” is what lets it be a long-lived personal agent — your assistant shouldn't be locked to one machine or vendor. It echoes the deployment philosophy: the agent's runtime is itself pluggable, from local all the way to a serverless sandbox:

The engine of evolution: the learning nudge

The loop has one catch: an LLM won't spontaneously reflect on “should I save what I just learned?” Hermes' fix is clever — after enough turns of working together, it appends a reflection prompt to the end of the chat (a nudge): “Did you just learn a repeatable procedure? Want to save it as a skill with skill_manage?” The nudge fires at the right moment, once per session, so it never nags.

What's neat is where it's injected: the nudge is a plain user message appended at the end, changing no system prompt and rebuilding no context. That's exactly to honor the iron rule — the prompt cache must not break. “Remind the agent to learn” sounds trivial, but underneath it's a careful tradeoff between driving evolution and never breaking the cache (details in ch.9).

What you're dealing with (I)

Before we dissect Hermes' design, let's see what we're dealing with. A single LLM call is, at heart, a stateless, token-based, probabilistic function: you hand it context, it emits the most likely next text. It's powerful, but ships with a set of hard, built-in constraints. This chapter covers the three that bite at the single-call level — attention & context (A), statelessness & nondeterminism (B), and the token representation layer (E). They're the “root cause” behind almost every later design: you'll see most of Hermes' choices work with these constraints, not fight them.

🔌 Analogy
Think of one call as consulting a brilliant expert with per-call amnesia: you write all the background on one sheet of paper, hand it over, they read only that sheet, give one answer, then forget everything. Ask again and you re-hand a sheet. Reading long text they care most about the start and end, and skim the middle; and they read in chunks (tokens), not letters — so “how many r's in strawberry” is exactly what they get wrong.

A · Attention & context: “fits in the window” isn't enough

🌍 Big picture
The model's attention is not uniform across the context. Both position and length strongly affect what it actually “sees” and how accurately. Treating context as a scarce resource is lesson one of agent design.

① Lost in the middle. Experiments consistently show the model attends most to the start and end, and least to the middle. A key instruction buried mid-context often goes “unseen.” The fix is direct — put key instructions at both the head and the tail; place the most relevant retrieved snippets at the edges, not the middle.

② Context rot. Even when the window isn't full, quality degrades with length: weaker instruction-following, more hallucination. So compression isn't only about cost / fitting — it's about quality: a tight, focused context usually beats “stuff everything in.” Same for retrieval — precision over recall, since irrelevant docs “distract.”

③ Length → latency & cost. Longer context means slower, pricier steps. An interactive agent needs a context budget and can't pile on history blindly. This directly motivates Hermes' iteration budget (ch.5) and context compression (ch.15).

⚠️ Counter-intuitive
“More context” isn't always better. Past a point, more context = worse answers. Less but focused > more but noisy.

B · Stateless & nondeterministic: no memory, no guaranteed repeatability

🌍 Big picture
Between two calls the model remembers nothing; and the same input need not yield the same output. Together: an “agent” is fundamentally orchestration around a stateless, nondeterministic function.

① Stateless. The model saves nothing between calls. All “memory” must be externalized (to files/DBs) or re-sent (packed into messages) every time. That's the root reason Hermes has skills, memory, a session store — the external hard drive for an amnesiac core (ch.11, 12).

② Nondeterminism. Even at temperature 0, the same input can produce different results. So you can't rely on reproducibility: design for retries, validation, idempotent tools, and use tolerant tests instead of literal assertions.

③ Autoregressive, no undo. The model generates token by token; once it starts down a wrong path, it tends to “commit and rationalize,” inventing its way forward. The fix: plan first, execute second, put “revision” in a separate pass, and give it room to reason (thinking tokens) before answering. This echoes Hermes' delegation / plan-execute split (ch.13, 14).

E · The token layer: it sees the world in chunks

🌍 Big picture
The model doesn't read characters, let alone numbers — it reads tokens (chunks). That makes it inherently unreliable at exact characters, counting, and arithmetic — work that should be handed to tools.

① Tokenization makes character/count/math weak. “How many r's in strawberry,” exact substring matching, multi-digit multiplication — the model often gets these wrong, because it sees chunks like str+aw+berry, not individual letters. Fix: don't let the model do exact math or character-counting; hand it to a calculator / code execution (ch.8, the tool system).

② Structured output is fragile. The model natively emits prose, not typed data. For reliable JSON you need function calling / JSON mode / grammar constraints + a validate-and-repair loop (ch.7).

③ Max-output truncation. Long output can be cut mid-structure. Fix: chunk and continue, design tasks against an output budget; don't expect one giant structure in a single shot.

What you're dealing with (II)

The previous chapter treated the model as a “part” and looked at its physics. This chapter shifts view: once you assemble that part into an agent that acts autonomously, over many steps, a new set of problems gets amplified. These are the “most painful” parts of agent engineering — truthfulness (C), instruction-following quirks (D), multi-step autonomy (F), and operations (G). Two are marked ⭐⭐: instructions and data are inseparable and error compounding — the source of nearly every agent incident.

🔌 Analogy
Now make that “amnesiac expert” do 20 steps in a row, each building on the last. Trouble: he's just as confident when fabricating (you can't tell guessing from knowing); he'll obey “instructions” smuggled into the notes (even if someone else slipped them in); and he loves to agree with you (push back and he flips). After 20 steps, small errors stack and the result may be unrecognizable.

C · Truthfulness: it fabricates, confidently

🌍 Big picture
The model is a reasoner, not a source of facts. It hallucinates, and it's poorly calibrated — “I'm sure” has no reliable link to “it's correct.” Treat its facts, IDs, and URLs as things to verify with tools, not accept at face value.

① Hallucination + poor calibration. It invents non-existent APIs, citations, numbers — in the same confident tone. Fix: ground with retrieval, demand citations, verify with tools. That's why Hermes leans on retrieval grounding (ch.12) and hands facts to tools.

② Context poisoning. Once a hallucination enters the history, it gets reinforced as fact downstream. Fix: don't feed the model's unverified claims back as fact; isolate and label untrusted content by source.

③ Knowledge cutoff. It only knows the world up to training. Fix: fetch fresh data with tools + inject the current date into the system prompt (exactly what Hermes does, ch.6).

⚠️ Key
“I'm sure” ≠ “it's correct.” The model's confidence and accuracy are decoupled. Don't take its word as fact — verify.

D · Instruction-following quirks: the heaviest trap

🌍 Big picture
The model can't reliably tell “instructions” from “data to be processed” — the root of prompt injection and the number-one agent-security problem.

① ⭐⭐ Instructions and data are inseparable. Tool output, web pages, retrieved docs, user files — anything entering the context can smuggle instructions that hijack the agent. The countermeasure is a combo: least-privilege tools, human-in-the-loop for dangerous actions, isolate/delimit untrusted content, and separate the planner (privileged) from the data processor (isolated). Hermes' gateway has two guards (ch.18); delegation does privilege isolation (ch.14).

② ⭐ Sycophancy. The model tends to agree with the user and flips when challenged. That makes “review/verify” agents dangerous — it'll happily say your buggy code “looks fine.” Fix: use an independent critic, adversarial framing; don't let the user's assertions poison verification (ch.14 background_review).

③ Generator-verifier gap. The model is often better at verifying than generating. So let another pass / another agent do the checking instead of one-shotting without review.

④ Prompt brittleness. Small changes in wording/format can change behavior a lot. Fix: treat prompts as code — version them, back them with an eval set (ch.22).

F · Multi-step autonomy: the agent's sorest spot

🌍 Big picture
Let the model run many autonomous steps and errors compound, goals drift. This is the root reason a narrow, specialized agent beats free-wheeling long autonomy.

① ⭐⭐ Error compounding. 95% reliable per step, 20 steps multiplies to ≈36%. Fix: keep loops short, verify each step, decompose, checkpoint. Hermes' delegation (ch.13) fights this with “short loop + decomposition.”

② ⭐ Long-horizon drift. In long tasks the agent forgets the goal, loops, spins. Fix: explicit goal tracking (todo), periodic “re-grounding” (restate goal + progress), step budget + loop-breaking (ch.9 todo, ch.5 iteration budget).

③ More tools = worse accuracy. More tools / overlapping semantics → worse selection. Fix: a small, orthogonal tool set, clear descriptions, grouping / dynamic loading. This echoes Hermes' narrow waist + Footprint Ladder (ch.8).

Project map · the narrow waist

Hermes is big — CLI, a messaging gateway, 20+ platforms, a TUI, a desktop app, cron, kanban… yet it has a clear shape: a narrow waist. The middle is a very narrow core (one shared core-tool list + one AIAgent), while both ends are wide — many front-ends on top, edge extensions (skills/plugins/MCP) below. Grasp this shape and you grasp most of Hermes' architecture: grow the core as little as possible; grow capability at the edges.

🔌 Analogy
Think of USB: the middle is one extremely narrow, extremely stable standard, while both ends connect countless devices — keyboards, drives, phones, printers. Precisely because the “waist” is narrow and stable, the ecosystem explodes at both ends. Hermes is the same: the waist (core tools + agent loop) stays narrow and stable; the ends (front-ends + extensions) grow freely.

One agent core, five front-ends

Every entry point — CLI, gateway, TUI, desktop, IDE (ACP) — ultimately drives the same AIAgent (run_agent.py). They're just different “shells” over a shared reasoning loop:

The “waist”: one shared core-tool list

The “waist” is concretely _HERMES_CORE_TOOLS — one core-tool list shared by all platforms. Its comment states the intent in one line:

Why so disciplined? Because every core tool ships in the tool schema of every API call — more tools, worse model selection (exactly ch.3's F “more tools = worse accuracy”). So the bar for a new core tool is very high.

Lifecycle of one conversation

From the moment you hit enter to the moment Hermes hands back a final answer, what actually happens? The answer is a synchronous main loop: every time the model "thinks a step" it may call tools, the tool results are appended back onto the message list, and the whole thing is fed back to the model — round after round, until the model stops calling tools and just answers. This loop is the heart of the agent. It lives in agent/conversation_loop.py and carries three hard constraints: interruptible, budgeted iterations, and strict message-role alternation. This chapter cracks that heart open.

🔌 Analogy
Picture one conversation as a factory assembly line: raw material (your message) enters, rides the belt past one station after another (each station = the model calling one tool), and each station drops its half-finished part back on the belt to continue. The line is synchronous — the next station won't start until the previous one finishes. It also ships two safety devices: a counter (stop after at most N stations, so it can't spin and burn money) and an emergency stop (you can halt it any time and it lets go instantly). Hermes' conversation loop is exactly this line, safety devices included.

Macro: one synchronous while loop

🌍 The big picture
Don't let Hermes' size scare you — the core of a conversation is an utterly ordinary synchronous while loop. Each turn does five things: check for interrupt → consume one unit of budget → call the model once → if there are tool_calls, run them and append results back to messages → if there are none, finish. All "memory" lives in the messages list and is resent in full every turn — because the model itself remembers nothing between calls (Ch. 2, B·statelessness). The loop never rebuilds context, only appends to the end, precisely to keep the prompt cache intact (Ch. 6).

Iteration budget: stopping runaway loops

At step 5, every model call first consume()s one unit of budget. That latch is guarded by IterationBudget (agent/iteration_budget.py), thread-safe, with a parent cap of 90 by default and each subagent capped independently at 50. Its two core methods are short enough to memorize:

agent/iteration_budget.py37-43 · 56-59
def consume(self) -> bool:
    """Try to consume one iteration.  Returns True if allowed."""
    with self._lock:
        if self._used >= self.max_total:
            return False
        self._used += 1
        return True

@property
def remaining(self) -> int:
    with self._lock:
        return max(0, self.max_total - self._used)

Why budget the loop at all? Because model errors compound turn over turn (Ch. 3, F·error-compounding): once the plan drifts, the model can call tools forever, circling without reaching the goal and draining your wallet. A bounded iteration count is the hard brake — it caps cost and latency (A·latency/cost) and backstops runaway loops. One thoughtful touch: execute_code's "programmatic tool calling" calls refund() to hand that unit back, so it doesn't eat the normal conversation budget.

Interruptible: halt any time

Every turn starts by checking _interrupt_requested — so who sets it? interrupt() (run_agent.py:2400-2440): it sets _interrupt_requested = True, records the interrupt message, and cascades the signal to in-flight tool threads and subagents so live operations abort fast. The classic trigger is the messaging gateway: when a session's agent is still running and you send a new message, the gateway calls running_agent.interrupt(new_message.text) — so it's cut off at the top of the next turn and pivots to your new instruction.

System prompt & caching

This is the book's core chapter. Almost every Hermes design ultimately answers one question: does it break the prompt cache? This chapter nails three things — how the system prompt is assembled in three tiers, why it stays byte-for-byte stable within a session, and how Anthropic's cache breakpoints cut multi-turn input cost by ~75%. One sentence is enough to remember: the system prompt is built once per session; the only exception is context compression.

🔌 Analogy
Think of the system prompt as a building's foundation and load-bearing walls: poured once at the start, relied on by the whole structure, and never knocked down and re-poured halfway through — that would collapse the building. Now view caching from another angle: it's like a prepaid “prefix pass” — the first turn pays full price to compute that long prefix and stores it in the upstream cache; every later turn, as long as the prefix hasn't changed by a single byte, only pays the balance (the few new lines), saving exactly the cost of recomputing the prefix. The moment you change the prefix mid-session (swap tools, reload memory, rebuild the prompt), the pass is void and every turn pays full price again.

Macro: three tiers ordered by stability

🌍 The big picture
The system prompt isn't a randomly concatenated blob — it's three tiers ordered by stability: stable (identity / tools / skills / environment, nearly constant), context (caller message + project context files, fixed for the session), and volatile (memory snapshot / user profile / timestamp, the most likely to change). The three are joined with \n\n into one string, built once at session start, cached on agent._cached_system_prompt, and reused verbatim every turn. Pushing the most volatile content to the very end means that even when it changes, only the tail of the string moves — the already-cached prefix is untouched.

Why does this discipline deserve the whole book bending to it? Because it directly sets the cost order of magnitude. A personal agent's conversations tend to be long — dozens to hundreds of turns, with that long prefix resent again and again. On a cache hit, a long conversation's marginal cost is almost just “the few new lines”; once the cache breaks, every turn pays full price from scratch. Chapter 1's iron rule — “the prompt cache is sacred” — finally lands here on concrete code and numbers.

The three tiers: more stable goes first

The assembly entry is build_system_prompt_parts() (system_prompt.py:113), which returns a three-key dict {stable, context, volatile}. The order is anything but arbitrary — the more stable goes first, the more volatile goes last:

Why insist on three tiers rather than mashing everything into one block? Because the cache is a prefix cache — it compares byte by byte from the start, and at the first differing byte everything from there on is invalidated. If a volatile timestamp or memory sat at the very front, every turn would change it and void the whole prefix. The essence of tiering is to physically separate the three kinds of content — “nearly never changes,” “fixed within the session,” and “may change each turn” — so that volatility only ever lands at the tail of the string.

The caching strategy: system_and_3 places 4 breakpoints

With a byte-stable prefix in hand, all that's left is to tell the upstream “where reuse may begin.” Hermes uses just one layout — system_and_3: one cache_control breakpoint on the system prompt and one on each of the last 3 non-system messages, 4 in total, all at the same TTL (5 minutes or 1 hour). It cuts multi-turn input token cost by ~75%:

def apply_anthropic_cache_control(api_messages, cache_ttl="5m", native_anthropic=False):
    messages = copy.deepcopy(api_messages)            # deep copy; never mutate the original
    marker = _build_marker(cache_ttl)
    breakpoints_used = 0
    if messages[0].get("role") == "system":
        _apply_cache_marker(messages[0], marker)      # breakpoint ①: the system prefix
        breakpoints_used += 1
    remaining = 4 - breakpoints_used
    non_sys = [i for i in range(len(messages))
               if messages[i].get("role") != "system"]
    for idx in non_sys[-remaining:]:               # breakpoints ②③④: last 3 non-system
        _apply_cache_marker(messages[idx], marker)
    return messages

Two details here are worth noting. First, the very first line does copy.deepcopy on the whole message list — it never marks the original; breakpoints are added only to the copy sent this turn, so the cached “clean” message sequence is never polluted. Second, all 4 breakpoints share one TTL (5m or 1h): a short chat is cheap at 5 minutes, a long session can switch to 1 hour to keep the prefix “warm” longer. Either way the placement is always the one layout: “system + last 3.”

Draw it out and it's obvious — the prefix (system) plus the 3 most recent tail messages, exactly 4 🔖:

Why ~75% and not 100%? Because a cache hit isn't free — it's a discount: the hit prefix bills at the cache-read price (usually a small fraction of full price), while the new tail still bills at full price. In a long multi-turn conversation the vast majority of tokens are the same fixed prefix resent every turn — each turn they pay the discounted price, so the conversation's input cost collapses to roughly a quarter. This also explains the reverse cost: change a single byte of the prefix mid-session and, from that turn on, the whole prefix loses the cache and is recomputed at full price — the deeper the conversation, the more that one slip hurts. “Don't touch the prefix” isn't fussiness; it's real money.

The last gate guarding the prefix: scan before injection

The context tier drops files like AGENTS.md / .cursorrules verbatim into the system prompt. The catch: these files may come from a cloned repo and may hide a prompt injection. Once it enters the system prompt it becomes part of the session's fixed prefix, and the user has no chance whatsoever to intercept. So Hermes scans before injecting:

agent/prompt_builder.py46-62 · simplified
def _scan_context_content(content: str, filename: str) -> str:
    # scan for injection BEFORE the context file enters the system prompt
    findings = _scan_for_threats(content, scope="context")

    return content

The comment states the reason plainly: the file would otherwise enter the system prompt verbatim and the user has no chance to intervene. On a hit it's swapped for a [BLOCKED: …] placeholder, keeping the threat out of the prefix. This maps straight onto an inherent LLM constraint: D·instr=data — the model can't tell “instructions” from “data,” so external text fed into the prompt must clear security first.

The gate is also restrained: it uses the "context" scope — covering classic injection, promptware / C2, and role-play hijack, but not the more aggressive strict-scope rules (SSH backdoor, persistence, exfil URL). The reason is in the comment: context files often come from a cloned repo (security research, infra docs), where strict scope would kill normal content. The scan must block injection without rejecting every project doc — that boundary is itself a design trade-off.

Hermes speaks to Anthropic, OpenAI Codex, Gemini, AWS Bedrock, and twenty-odd OpenAI-compatible backends. Yet its core conversation loop knows just one message format end to end — OpenAI-style messages. This chapter shows how that unified abstraction works: a transport registry dispatches by api_mode, four adapters translate the unified format into each backend's dialect, reasoning is replayed verbatim across turns, plus one invariant that runs through it all — strict role alternation.

🔌 Analogy · simultaneous interpreters + a universal adapter
Picture the core loop as a commander who speaks only one language. Facing generals from all over the world (providers), he doesn't learn twenty tongues — he hires a team of simultaneous interpreters (adapters): the commander always issues orders in "standard speech" (OpenAI messages), and the interpreter translates on the spot into each dialect (Anthropic blocks / Codex responses / Gemini parts / Bedrock converse); replies are translated back. The universal adapter (the transport registry) decides "which interpreter for this general" — look up api_mode, fetch the matching transport. Add a new country and you just hire one interpreter and register one adapter — the commander changes not a single word.
🌍 The big picture · one set of messages, many backends
The core loop (ch.5) always works on one OpenAI-style messages list: {"role": "system/user/assistant/tool", ...}. Every provider difference — auth, field names, reasoning shape, tool-schema dialect — is held at the transport + adapter layer and never seeps into the core. Outbound, build_api_kwargs picks a transport by api_mode to translate the unified messages into each backend's request; inbound, normalize_response normalizes each backend's reply back into the same assistant message (reasoning included). This is one concrete landing spot for the ch.4 "narrow waist": thin core, thick edges.

Where dispatch begins: the transport registry

Each api_mode maps to a transport class. The registry loads via lazy discovery — it scans and imports all transport modules only on first fetch, then hits the cache:

agent/transports/__init__.py21-46 · excerpt
def register_transport(api_mode, transport_cls):
    # the four transport modules each call this at file end to register
    _REGISTRY[api_mode] = transport_cls

def get_transport(api_mode):
    global _discovered
    if not _discovered:
        _discover_transports()              # lazy: import all transports on first use
    cls = _REGISTRY.get(api_mode)
    if cls is None:
        _discover_transports()              # re-scan on miss; tolerate out-of-order imports
        cls = _REGISTRY.get(api_mode)
    if cls is None:
        return None                        # let callers fall back to the legacy path
    return cls()

The four transports — anthropic_messages / codex_responses / chat_completions / bedrock_converse — call register_transport(...) at the bottom of their modules to hook themselves into _REGISTRY. Note that get_transport re-scans on a miss: tests or out-of-order imports may leave the registry half-populated, so "discover again if not found" guarantees a valid api_mode is always resolvable. Only then does it return None, letting the caller fall back gracefully — a hook left for gradual migration.

One asymmetry is worth noting: there are four transports (including bedrock), but Gemini is not among them. Gemini uses client-layer substitution — when the base_url points at native Gemini, GeminiNativeClient directly replaces the underlying OpenAI client (agent_runtime_helpers.py), while its api_mode stays chat_completions. So strictly, the translation layer is 'four transports + one client swap', and the adapter list (anthropic / codex_responses / gemini_native / bedrock) doesn't fully overlap the transport list — yet all of this stays transparent to the core loop.

Outbound dispatch: build_api_kwargs routes by api_mode

What actually turns unified messages into "one backend's request params" is build_api_kwargs. It is a set of hard branches keyed on api_mode, each fetching the right transport and calling its build_kwargs:

def build_api_kwargs(agent, api_messages):
    if agent.api_mode == "anthropic_messages":

        return agent._get_transport().build_kwargs(model=..., messages=api_messages, ...)

    if agent.api_mode == "codex_responses":          # /responses endpoint, with xAI/GitHub routing
        return agent._get_transport().build_kwargs(...)

    ...                                              # else: default chat_completions (most common)

The thing to grasp: the core loop has no idea what "Anthropic looks like." It only calls build_api_kwargs, which routes the work to the matching transport / adapter by api_mode. The Bedrock branch even bypasses the OpenAI client entirely, with the adapter making boto3 calls directly — yet that's transparent to the core. Each new backend is just one more if branch + an adapter here, with not a line changed in the loop itself.

Continuity across turns: reasoning's three stores

"Thinking" (reasoning) models bring a tricky demand: this turn's reasoning must be handed back verbatim next turn, or some providers return HTTP 400 outright. Hermes splits reasoning into three fields, each handled differently:

    msg["reasoning_content"] = reasoning_text or " "

# ③ reasoning_details: passed back unmodified to keep reasoning continuous

    # maintain reasoning continuity across turns.
    msg["reasoning_details"] = preserved      # carries signature/encrypted opaque fields

The three fields each do their job: reasoning is the internally kept thought text; reasoning_content is the field that DeepSeek-v4 / Kimi-style thinking models strictly require on replay — miss it and the next turn hits a 400, "The reasoning_content in the thinking mode must be passed back to the API", so even with no captured text Hermes pads a single space (satisfying the non-empty check without fabricating reasoning); reasoning_details is passed through unchanged — the comment says it plainly: Pass reasoning_details back unmodified so providers ... can maintain reasoning continuity across turns. Its opaque signature / encrypted_content fields would break the reasoning chain if a single byte changed. This maps to the LLM constraint G·reasoning tokens not persisted — the model can't remember how it reasoned last turn, so we must carry the reasoning trace verbatim back into visible context.

Outbound to inbound: one complete data flow

1unified messages (OpenAI style — the only format the core loop knows)
2repair_message_sequence_with_cursor — fix strict role alternation before every API call
3build_api_kwargs fetches a transport by api_mode
4adapter translates into each dialect (Anthropic blocks / Codex input / Gemini parts / Bedrock converse)
5provider API returns a response
6normalize_response normalizes back to a unified assistant message (with reasoning's 3 fields)
7append into messages — back to the core loop, on to the next turn

Step 2's role-alternation repair is the invisible gatekeeper of the whole chain. Providers broadly require messages to strictly alternate (no two same-role in a row, no orphan tool messages); violate it and you get an empty response and a retry. repair_message_sequence_with_cursor runs before every API call: it drops unpaired orphan tool messages and merges consecutive user messages. It tidies the live messages in place right before sending (messages[:] = merged, so persistence / SessionDB see the repair), doing only minimal surgical cleanup and never aggressively rebuilding context — avoiding needless history rewrites that would shatter the cache (ch.6).

Tool system

Tools are how an agent "acts" — read files, run a terminal, search the web, drive a browser. This chapter shows how a tool goes from one Python file to a capability the model can call: registry auto-discovery, check_fn service gating, toolset-filtered schema assembly, a unified JSON return + error sanitizing, and execute_code refunding budget. It also answers a question that runs through the whole book — why adding a core tool is treated so carefully: because every tool's schema is sent on every single API call.

🔌 Analogy · airport security + gate assignment
Picture the toolset as an airport. For a flight (tool) to board, it clears two gates: security (check_fn) — a flight without its prerequisites (API key, playwright, docker) never even appears on the board; and gate assignment (toolset filtering) — only tools in this platform's enabled toolset make it onto the floor. The board the passenger (model) sees (the tool schema) is reprinted and sent to the tower every time — so the more flights, the costlier each transmission. That's why "opening another core route" is weighed so carefully.
🌍 The big picture · capability at the edges, a narrow-waist core
Hermes keeps its core toolset deliberately narrow (ch.4's "narrow waist"). The reason is hard: every tool's JSON schema ships with every API call — more tools mean higher fixed overhead and more crowded attention. So new capability follows the "Footprint Ladder": extend existing > CLI command + skill > check_fn-gated tool > plugin > MCP server > only last, a new core tool. Registering a tool takes two things: write tools/your_tool.py calling registry.register(name, toolset, schema, handler, check_fn=...), then put the tool name into a toolset — but "should it go in the core" is always the most expensive decision.

Gating: check_fn makes a tool "vanish when prerequisites are unmet"

When assembling the schema list sent to the model, the registry runs each tool's declared check_fn. A tool returning False is skipped outright — never entering the schema; when its condition is unmet, its footprint is zero:

tools/registry.py337-384 · simplified
def get_definitions(self, tool_names):
    result = []
    check_results = {}                                # per-call cache within one assembly pass
    for name in sorted(tool_names):
        entry = entries_by_name.get(name)
        if entry.check_fn:                            # tool declared a prerequisite
            if entry.check_fn not in check_results:
                check_results[entry.check_fn] = _check_fn_cached(entry.check_fn)  # cached ~30s
            if not check_results[entry.check_fn]:
                continue                            # unmet -> not in schema (zero footprint)
        schema_with_name = {**entry.schema, "name": entry.name}
        result.append({"type": "function", "function": schema_with_name})
    return result

Two bits of cleverness. First, the check_fn result is cached by _check_fn_cached for about 30 seconds — because checks like "probe docker / probe playwright / probe an API key" cost something, the 30s TTL lets hermes tools enable foo take effect in near-real-time without re-probing on every call. Second, gating keeps the toolset stable for a session: Home Assistant tools appear only when a token is configured, otherwise they don't even occupy a schema — which serves ch.6's caching rule (the tool schema is part of the prefix too, so never swap toolsets mid-session). One level up, get_tool_definitions (model_tools.py:276) first filters by enabled/disabled toolsets, and the result is memoized on config.yaml's mtime+size — no recompute if config is unchanged.

Dispatch: a unified JSON return + error sanitizing

The model returns a tool_call; dispatch finds the handler by name and runs it. Every handler returns a JSON string, and every exception is uniformly wrapped into {"error": ...} — after being sanitized:

tools/registry.py390-416 · simplified
def dispatch(self, name, args, **kwargs):
    entry = self.get_entry(name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {name}"})
    try:
        return entry.handler(args, **kwargs)         # handler must return a JSON string
    except Exception as e:
        raw = f"Tool execution failed: {type(e).__name__}: {e}"
        # sanitize: framing tokens / CDATA / fences in the exception
        # string must not reach the model as structural noise
        sanitized = _sanitize_tool_error(raw)
        return json.dumps({"error": sanitized})

The point here isn't "catch the exception" — it's sanitizing. A tool's output — success result or error string alike — is data to the model, not instruction. Yet an exception message might accidentally carry the model framework's special tokens, CDATA, or code fences; fed back verbatim, the model could misparse them as structural signals. _sanitize_tool_error strips those so the error stays honest data. This maps to D·instr=data — the model can't tell "instructions" from "data," so tool output fed back must be disinfected first.

Budget: execute_code refunds an iteration

Ch.5 covered the iteration budget (parent 90 / subagent 50), where each tool turn spends one slot. But there's an exception — execute_code (programmatic tool calling) is a cheap RPC-style call that shouldn't eat the budget. So if and only if every tool_call this turn is execute_code, one slot is refunded:

agent/conversation_loop.py4117-4122 · excerpt
# Refund the iteration if the ONLY tool(s) called were execute_code.
_tc_names = {tc.function.name for tc in assistant_message.tool_calls}
if _tc_names == {"execute_code"}:
    agent.iteration_budget.refund()

Note the strict equality == {"execute_code"}: if any other tool is mixed in this turn, there's no refund. This ensures refunds only happen on "purely programmatic" turns and can't be gamed by smuggling other tools inside execute_code to live forever. Budget (ch.5), gating (this chapter), and caching (ch.6) together let an agent that calls tools freely stay both powerful and under control.

Learning engine & skills

Hermes's signature feature is self-improvement: across sessions it gets better at understanding you and at doing your kind of task. This chapter covers the entry point of that learning loop — every ~10 turns (a configurable default), after a turn ends and the response is already delivered to you, Hermes forks a background review agent that replays the conversation and asks itself, "was there a skill or memory worth saving this turn?" The crucial line: all of this happens in the background, and the main conversation and prompt cache are never touched, not by a single byte.

🔌 Analogy · an after-hours debrief / an apprentice's notes
Think of a diligent apprentice. While working during the day (the main conversation) he is fully focused on finishing the job — never stopping to journal, which would interrupt the task at hand. But after hours (the response already delivered) he reviews the day's tickets, distilling "how this class of job is done" into a manual (a skill) and jotting down "who this client is and what they like" into a memo (memory). Next morning he shows up with an updated manual, yet today's workflow was never once interrupted. Learning happens "at another time, in another set of hands" — exactly why the cache stays intact.
🌍 The big picture · procedural vs declarative memory
Hermes splits "what it learns" into two kinds: skills are procedural — "how to do this class of task" (SKILL.md + scripts/templates/references), executable, reusable, loaded on demand; memory is declarative — "who the user is, what the current state is" (ch.11). Both are produced by the same background review fork. Skills sink to the edges (ch.4's narrow waist: capability stays out of the core, loaded on demand), so they neither crowd the per-call tool schema nor blow up the context.

The skill/memory boundary: which one to write to

What goes into a skill versus a memory? The background review's prompt draws that line sharply — this is the verbatim instruction actually fed to the review fork:

agent/background_review.py233-237 · excerpt (verbatim review prompt)
Memory captures 'who the user is and what the current situation
and state of your operations are'; skills capture 'how to do
this class of task for this user'. When they complain about how
you handled a task, the skill that governs that task needs to
carry the lesson.

In a sentence: memory = who / what-state (compact, injected every turn), skill = how to do this class of task (an executable manual loaded on demand). When the user complains you handled a task poorly, the right move isn't "log a memory" — it's to update the skill that governs that class of task, so the lesson lands in "how to do it next time."

The trigger: nudge counts, but acts only after the response

Learning is driven by a counter: it increments each turn and sets a boolean at the threshold. The key: it injects no text into the main conversation — it's just a flag, consumed after the response has gone out:

agent/turn_finalizer.py432-456 · simplified
# Decide NOW whether to trigger a skill review, based on this turn's iterations
_should_review_skills = False
if (agent._skill_nudge_interval > 0
        and agent._iters_since_skill >= agent._skill_nudge_interval
        and "skill_manage" in agent.valid_tool_names):
    _should_review_skills = True
    agent._iters_since_skill = 0

# Background review — runs AFTER the response is delivered, never competing
# with the user's task for model attention
if final_response and not interrupted and (_should_review_memory or _should_review_skills):
    agent._spawn_background_review(
        messages_snapshot=list(messages),       # a snapshot, not live messages
        review_memory=_should_review_memory,
        review_skills=_should_review_skills,
    )

Note the comment: runs AFTER the response is delivered so it never competes with the user's task for model attention. After a nudge fires, no text enters the main conversation — it's only the boolean _should_review_skills = True; the real review waits until the response is out and is consumed in turn_finalizer. What's handed to the fork is list(messages) — a snapshot, leaving the original conversation untouched.

Why the cache survives: fork a separate agent

The boolean is consumed by a separately forked review agent. Its module docstring pins the invariant:

This is the whole secret of how "self-improvement" coexists with "the cache is sacred": the review runs in a daemon thread + a forked AIAgent, replaying a conversation snapshot, asking whether to save, writing straight into the skill/memory stores. The main conversation's message sequence, system prompt, and upstream cache — all unchanged throughout. The write happens elsewhere; a new skill only enters the stable prefix next session.

Two guardrails: provenance gating + deferred invalidation

This auto-learning has two key guardrails. First, provenance gating: only skills created by the background review fork are marked "agent-created" and fall under the curator (ch.10) — skills you made by hand are never touched:

This is the "cache-aware slash command" pattern pinned in AGENTS.md. The throughline: the learning loop never breaks the cache. Curator: the skill gardener

In ch.9, the background review keeps creating new skills. A problem follows: won't the skill library grow into a mess, stuffed with one-off skills used exactly once? That's what the Curator (the skill gardener) solves — it periodically patrols agent-created skills in the background and auto-demotes / archives the long-unused ones. But it has an iron rule: archive only, never delete, and the whole process never touches the main conversation's prompt cache. This chapter is also a model of "data safety + auxiliary-model isolation."

🔌 Analogy · a librarian + a recoverable holding stack
Picture the skill library as a real library. The librarian (curator) does a round every so often, moving books nobody has borrowed in a while to a "holding area" (stale), and after even longer into the basement stacks (archived). But he never burns books — anything in the basement can be retrieved. Books a reader has pinned stay on the shelf, untouched by the patrol. Crucially, the librarian works in a separate office (a forked agent), never interrupting you reading in the main hall (the main conversation).
🌍 The big picture · self-maintaining, yet never destructive
The Curator's design is captured in one phrase: self-maintaining, yet never destructive. It only touches skills auto-created by the background review fork (marked created_by=agent) — your hand-made ones (never marked), hub-installed, and protected built-ins are off-limits; it never deletes — the most aggressive move is archiving (recoverable); pinned skills bypass all auto-transitions; and it runs on the auxiliary model in a separate session, never touching the main conversation's prompt cache. These four invariants are exactly the safety boundary between "let the agent evolve freely" and "data is never lost, the cache is never broken."

A deterministic state machine: active → stale → archived

The core of demotion is a purely deterministic state machine — calling no LLM, looking only at a skill's "latest real activity" timestamp:

    if row.get("pinned"):
        continue                                  # pinned bypasses every transition

        archive_skill(name)                       # 90 days idle -> archive (recoverable)
    elif anchor <= stale_cutoff and current == ACTIVE:

All three transitions hinge on anchor (latest activity): 30 days idle is active → stale, 90 days idle is stale → archived, and any fresh use is stale → active. The whole pass makes not a single LLM call, running purely on telemetry timestamps (fed by ch.9's skill_usage), making it both cheap and predictable.

Memory: MEMORY / USER

Memory lets Hermes remember across sessions "who you are and what you prefer": MEMORY.md is the agent's personal notebook, USER.md its profile of you. But a sharp tension hides here — memory is written at any time (the preference you just stated, the lesson the agent just learned), yet the system prompt must never change mid-session (ch.6). Hermes satisfies both "write anytime" and "never break the cache" with two carefully designed injection paths: a frozen snapshot + appending only to the current user message's copy.

🔌 Analogy · an assistant's notebooks + a Polaroid snapshot
Picture an assistant with two notebooks on the desk: MEMORY.md records "what this project deploys with, what pitfalls we hit," USER.md records "this boss likes brevity, hates rambling." Each morning he takes a snapshot (the frozen snapshot) and pins it where it's most visible, and looks at that all day — even if he jots new things into the notebooks midday, he doesn't swap the snapshot, to avoid distracting himself by re-reading. When he needs an old record temporarily, he doesn't alter the pinned snapshot — he copies a sticky note onto the current page (appending to a copy). Both paths never touch that sacred snapshot.
🌍 The big picture · declarative memory, written durably, injected cache-safe
Memory is declarative ("who / what-state"), distinct from ch.9's procedural skills ("how to"). MEMORY.md (the agent's notes, ~2200-char cap) and USER.md (the user profile, ~1375-char cap) live under $HERMES_HOME/memories/. The fix for the core tension is read/write separation: writes hit disk anytime (durable); but what enters the system prompt is the snapshot frozen at session start, and real-time recalled memory is appended only to the current user message's API copy. Both injection paths bypass "changing the system prompt mid-session" — that's the whole way memory honors "the cache is sacred."

Path one: a frozen snapshot enters the system prompt

Memory enters the system prompt's volatile tier (the lowest of ch.6's three tiers). But injecting live memory directly would change the system prompt with every memory you write, shattering the cache. Hermes's fix — what's injected is always the snapshot frozen at session start:

tools/memory_tool.py567-578 · excerpt
def format_for_system_prompt(self, target):
    """Return the frozen snapshot for system prompt injection.
    This returns the state captured at load_from_disk() time, NOT the
    live state. Mid-session writes do not affect this. This keeps the
    system prompt stable across all turns, preserving the prefix cache."""
    block = self._system_prompt_snapshot.get(target, "")
    return block if block else None

The docstring pins the intent: NOT the live state. Mid-session writes do not affect this. This keeps the system prompt stable across all turns, preserving the prefix cache. _system_prompt_snapshot is captured at session-start load_from_disk(); no matter how many entries you write via the memory tool afterward, this snapshot doesn't change → the system prompt stays byte-stable → the prefix cache holds. The snapshot refreshes only at the next session start, or at a context-compression boundary (ch.6's sole rebuild moment).

Path two: real-time recall appends only to the current user message's copy

So what about recalling an old memory mid-session? (e.g. relevant context fetched by an external provider.) The answer: it doesn't enter the system prompt — it's appended only to the "send copy" of the current user message:

agent/conversation_loop.py738-758 · simplified

The key is line three, api_msg = msg.copy() — injection happens only on the copy sent to the API, as the source comment states plainly: the original message in `messages` is never mutated, so nothing leaks into session persistence. The recalled content is wrapped by build_memory_context_block in a <memory-context> fence (marking it "recalled reference data, not a new user instruction") and appended to the end of the current user message. So: all prior turns stay byte-identical (the cache prefix is intact), the system prompt is unchanged, and the original messages aren't polluted or persisted.

Cross-session search: FTS5

You've chatted with Hermes hundreds of times, and one day you ask, "what was that deployment plan we discussed before?" — and it remembers, across sessions. That's cross-session search. But it has a counter-intuitive design: zero LLM, zero cost. It calls no model to "understand" or "summarize" — it queries a local SQLite FTS5 full-text index directly and returns the original messages verbatim for the agent to read itself. And recall doesn't break the prompt cache.

🔌 Analogy · a library card catalog
An old library's card catalog: as each book is shelved, the librarian copies its keywords into the card drawers (index on write). To find a book, you don't hire a scholar to read every book and tell you the gist (that's LLM summarization — slow and expensive) — you flip through the cards, sorted by relevance, and get the book's page numbers and excerpts to read yourself. Hermes's cross-session search is that card drawer: the FTS5 index is built as messages are written, and a search returns originals directly, never disturbing any model.
🌍 The big picture · index on write, recall verbatim, zero LLM
Three things form this chain: ① index on write — each message INSERTed into SQLite is synchronously fed by a trigger into the FTS5 inverted index (content + tool_name + tool_calls); ② FTS5 retrieval — search uses MATCH + BM25 relevance ranking + snippet() highlighting, with a trigram substring fallback for CJK; ③ verbatim append — the matched original messages are appended as a tool-result message at the end of the conversation for the agent to read. Not a single LLM call — which both saves money and (being just an appended tool message) doesn't touch the system prompt or break the cache.

Index on write: an AFTER INSERT trigger

No separate "build the index" step is needed — the moment a message is written into the messages table, a SQLite trigger synchronously sends it into FTS5:

hermes_state.py612-621 · excerpt
CREATE VIRTUAL TABLE messages_fts USING fts5(content);

CREATE TRIGGER messages_fts_insert AFTER INSERT ON messages BEGIN
    INSERT INTO messages_fts(rowid, content) VALUES (
        new.id,
        COALESCE(new.content, '') || ' ' ||
        COALESCE(new.tool_name, '') || ' ' ||
        COALESCE(new.tool_calls, '')
    );
END;

Two bits of cleverness. First, rowid = new.id — the FTS index's row id is messages.id, so a hit can JOIN straight back to the original message. Second, what's indexed isn't just content but content + tool_name + tool_calls concatenated — so even "which tool was called when" is searchable. Index-on-write means: no offline indexing needed, and what you just said is retrievable a second later.

Retrieval: FTS5 MATCH + BM25 + snippet

The search itself is one pure SQL statement, fully using FTS5's features — full-text match, relevance ranking, highlighted excerpts:

hermes_state.py3532-3579 · simplified
SELECT m.id, m.session_id, m.role,
       snippet(messages_fts, 0, '>>>', '<<<', '...', 40) AS snippet,
       m.content, m.timestamp, s.source
FROM messages_fts
JOIN messages m ON m.id = messages_fts.rowid    -- rowid JOINs straight back
JOIN sessions s ON s.id = m.session_id
WHERE messages_fts MATCH ?                    -- full-text match
ORDER BY rank                                -- BM25 relevance ranking
LIMIT ? OFFSET ?

Three FTS5 features each do their job: MATCH does inverted full-text matching (AND/OR/NOT/phrase/prefix); ORDER BY rank is FTS5's built-in BM25 relevance ranking; snippet() wraps hit terms in >>>…<<< as a highlighted excerpt of up to 40 tokens. For space-less CJK there's a second messages_fts_trigram table (trigram tokenizer) for substring fallback, dropping to LIKE for 1–2 characters. The whole retrieval involves no model at all.

Delegation & subagents

When a task produces tons of intermediate work (reading dozens of files, running a chain of commands), stuffing it all into the main conversation floods the context. Delegation (delegate_task) solves this: hand the subtask to an independent subagent — it has its own conversation, its own terminal, its own toolset, and returns only the final summary to the parent; the intermediate tool results never enter your context window. This chapter opens "scaling & isolation."

🔌 Analogy · outsourcing to an independent team
You're the project manager. A research task requires sifting a hundred documents — you don't haul all hundred onto your own desk to read (that buries it = floods the context). You outsource to an independent team: hand them a self-contained brief (the subagent knows nothing about your conversation history), they work in their own office (independent context + terminal), and finally give you a conclusion summary. The hundred documents they read and the drafts they tried never appear on your desk.
🌍 The big picture · context isolation, return only the summary
Delegation's core value is context isolation. The subagent gets an ephemeral_system_prompt (just goal + context, not the parent's conversation history), plus skip_context_files / skip_memory — it doesn't read the parent's AGENTS.md or the shared MEMORY.md. It runs to completion in its own conversation, and only the final response is returned to the parent as a summary. So the parent's context window stays clean: neither flooded by intermediate tool results nor polluted by the subtask's trial-and-error. This is another route against "limited context" — ch.15 uses compression, this chapter uses isolation.

A top-level, model-issued delegation always runs in the background: it returns a delegation_id immediately, and the subagent runs on a background daemon thread. Review & verification

Ch.13 gave you a "context-isolation hammer" — delegate_task. But complex workflows like "plan first, execute later" and "have an independent party review" are not built into Hermes's delegation layer. That is exactly this chapter's notable trade-off: Hermes does not bake planning/execution/review into the core — it uses skills to orchestrate delegate_task's isolation, building "generator-verifier separation" at the edges. The core provides just one primitive (an isolated subagent); the complexity lives in skills — the narrow-waist philosophy (ch.4) applied to multi-agent collaboration.

🔌 Analogy · peer review / third-party audit
No author should review their own paper — they have blind spots about their own oversights. Academia's answer is peer review: find someone who wasn't involved in the writing, give them only the paper (not your draft notes, not your rationalizations), and let fresh eyes find the flaws. Code is the same: implementer, reviewer, fixer are three separate people, each with a fresh context. Hermes implements this "independent verification" via delegate_task's context isolation — the reviewer gets only the diff, with no shared context with the implementer.
🌍 The big picture · generator-verifier separation rests on "independent context"
One principle governs this chapter: no agent should verify its own work (No agent should verify its own work). Why? A model has a confirmation bias about what it just generated, and is prone to sycophantic agreement. The countermeasure is verifying with a fresh context — the verifier doesn't know what the generator "wanted to prove," it only sees the result. Hermes doesn't make this a built-in flow of the delegation tool — instead three skills each play a part: plan (pure planning), subagent-driven-development (execution + two-stage review), requesting-code-review (independent review + auto-fix). The execution/review two of them call delegate_task's isolation to manufacture a "fresh context" (plan only produces a plan and doesn't delegate).

Context compression

Ch.6 laid down the iron rule: the prompt cache is sacred, and the only exception is context compression. This chapter is about that exception. When a conversation grows close to the model's context window, Hermes summarizes and compresses the early history into a shorter span. That step necessarily rewrites the cached prefix — the cache is bound to be voided. So it's designed to fire only as a last resort: spend one "cache reset" to buy back room to keep talking. This is the one place "the cache is sacred" yields, and the core weapon against "limited context" and "context rot."

🔌 Analogy · compressing meeting minutes into key points
A three-hour meeting has tens of thousands of words of verbatim notes — keep going and a newcomer can't possibly read the backstory. The smart move: keep the recent discussion (the tail) and the original agenda (the head), and compress the long middle into a page of structured key points (what was done, what was decided, what's left). After compression the meeting continues, but the minutes are a new version — exactly the cost of "cache invalidation." Hermes only does this when the minutes are about to overflow, never touching them otherwise.
🌍 The big picture · spend one cache reset to buy back room
Compression's logic chain: ① fire only near the limit — by default it considers compaction at 50% of the context window, with anti-thrashing guards; ② protect head and tail, summarize the middle — keep the original task framing (head) and recent context (tail), and use an auxiliary model to summarize the middle turns into structured key points; ③ rebuild the prefix — compression rewrote history, so it must _invalidate_system_prompt() to clear the cache and _build_system_prompt() to rebuild. That third step is the proof of "the cache's only exception." It treats A·lost-in-the-middle (limited context) and F·error accumulation (context rot).

When it fires: only near the limit, and anti-thrashing

Compression isn't done anytime — it fires only near the window, and guards against "compress a tiny bit then compress again" thrashing:

agent/context_compressor.py964-984 · simplified
def should_compress(self, prompt_tokens=None) -> bool:
    # fires only at the token threshold = 50% of the context window (threshold_percent=0.50)
    tokens = prompt_tokens if prompt_tokens is not None else self.last_prompt_tokens
    if tokens < self.threshold_tokens:
        return False
    # anti-thrashing: if the last two compressions each saved <10%, skip — avoid a loop that removes 1-2 msgs each time
    if self._ineffective_compression_count >= 2:
        return False
    return True

Two gates: first, don't compress until tokens hit the threshold (default 50% of the window, rising to 85% for small models near the edge) — normally "use up" the whole context first; second, anti-thrashing: if the last two compressions each saved less than 10%, there's nothing left to squeeze, and compressing again would spiral into a "remove 1-2 messages each time" loop, so it skips. Together these are "last resort only" — because each fire voids the cache once, an expensive cost.

The cache's only exception: rebuild the system prompt after compression

Here are the book's key three lines — why compression is the iron rule's only exception is proven right here:

new_system_prompt = agent._build_system_prompt(system_message)  # rebuild
agent._cached_system_prompt = new_system_prompt      # write back the new prefix

Normally (ch.6), once _cached_system_prompt is built it stays byte-stable, reused all session to hit the prefix cache. Terminal backends

For an agent to "act" it must run commands — but where? Your local machine, a Docker container, a remote server, or a serverless platform like Modal? Hermes's answer: one terminal tool, backed by pluggable backends. The model always calls a single terminal; whether a command lands on local / docker / ssh / modal / singularity / daytona is dispatched transparently by a unified abstraction. This lets one agent run across all environments, and lets serverless backends spin up on demand and release when done to save money.

🔌 Analogy · a universal remote
A universal remote's buttons are always "on / off / volume" (unified interface), but it can drive a TV, an AC, a speaker (multiple backends). You don't learn a new button layout per device — the remote internally translates "volume+" into each device's IR code. Hermes's terminal is that universal remote: the model always presses "run this command," and the BaseEnvironment abstraction translates it into local's subprocess, docker's exec, ssh's remote command, modal's serverless call. Swap the device by swapping the backend — the remote's buttons don't change at all.
🌍 The big picture · a unified abstraction, backend differences at the edges
Six execution backends — local / docker / ssh / singularity / modal / daytona — share one interface, BaseEnvironment (ABC). Each backend implements just two abstract methods, _run_bash() and cleanup(); the unified execute() (session snapshot, CWD tracking, interrupt, timeout) is provided by the base class and reused by all. A factory, _create_environment, picks the backend by the TERMINAL_ENV config. This is the narrow waist (ch.4) at the execution layer: the core knows only one terminal tool, and the differences of six environments are all caged in edge BaseEnvironment subclasses — add a new backend, and the core doesn't change a line.

The unified interface: the BaseEnvironment ABC

All backends' "common contract" is an abstract base class — subclasses fill just two holes, the base class handles the rest:

    Subclasses implement _run_bash() and cleanup(). The base class
    provides execute() with session snapshot sourcing, CWD tracking,
    interrupt handling, and timeout enforcement."""

    def _run_bash(self, cmd_string, ...) -> ProcessHandle:
        # subclass must implement: spawn a bash process in its own backend
        raise NotImplementedError

    @abstractmethod
    def cleanup(self):
        # release backend resources (container/instance/connection)
        ...

Note the division: subclasses implement only _run_bash() (how to spawn a process) and cleanup() (how to release resources); while execute() — session snapshot restoration, CWD tracking, interrupt handling, timeout enforcement — these cross-backend common flows are provided uniformly by the base class, reused by all six backends. So the docker backend cares only about "how to run bash in a container," the ssh backend only about "how to run remotely," and neither rewrites the common session-state management.

So what exactly is that "session snapshot"? Take the local backend — the design is elegant: every execute() spawns a brand-new bash process (spawn-per-call) that exits the moment the command finishes. But then wouldn't the env vars exported by the previous command, and the directory it cd'd into, be lost for the next one? The base class's "session snapshot" is exactly for this: after each command it snapshots env vars into a file and writes the current working directory to a file too; before the next command runs, it reads them back and sources them in. So a string of stateless, short-lived bash processes is woven into a session "with memory" — precisely the countermeasure for constraint B (statelessness) at the shell-execution layer: the underlying processes are stateless, and continuity is rebuilt from external file snapshots.

Gateway & platform adapters

A single Hermes process can send and receive across Telegram, Discord, Slack, Matrix, Feishu, WeChat… twenty-plus platforms at once. How? The core knows only one kind of message — each platform gets a "translator" (adapter) that turns its motley protocol into one unified event, and the core gateway's main path barely touches any specific platform.

🔌 Analogy · UN simultaneous interpretation
Delegates speak different languages (Telegram's Update, Discord's Gateway events, a line of IRC text…), but the floor runs in one working language. Each language gets an interpreter (adapter) who renders it into the working language and translates resolutions back. The chair (the core gateway) only needs to understand the working language, never the two dozen native tongues.
🌍 Macro · unified abstraction, platform differences at the edges
The core defines two unified abstractions: MessageEvent (a normalized "one message") and BasePlatformAdapter (the adapter contract). Gateway message guards

When the agent is busy on a long task and the user sends another message — what now? By default it queues, waiting for the agent to finish. But a few commands must never queue: /stop (interrupt), /approve (the agent is blocked waiting for it). If they queue, they either leak as conversation text and get discarded, or deadlock outright. Hence the gateway's "two guards + control-command bypass."

🔌 Analogy · the ER triage desk
Ordinary patients check in and queue (messages into _pending_messages), waiting to be called. But "cardiac arrest" (/stop) must jump the queue straight to resuscitation; and "the lab result the attending is waiting on" (/approve), if it also queues, leaves the doctor waiting forever, stuck. The triage desk must recognize these "must-not-queue" cases and let them through.
🌍 Macro · two sequential guards, control commands bypass
While the agent runs, a message passes two sequential guards: ① the adapter layer (queue if busy), ② the gateway runner layer (dispatch slash commands). Ordinary chat queues dutifully; but any resolvable slash command must bypass both and dispatch inline — otherwise it gets discarded as conversation text (empty response) or deadlocks.

Why queueing breaks things: the zero-char response

The bypass predicate's docstring spells out the consequence:

hermes_cli/commands.py379-399 · simplified
def should_bypass_active_session(command_name):
    """Return True for any resolvable slash command.
    Queueing is always wrong for a recognized slash command
    because the safety net in gateway.run discards any command
    text that reaches the pending queue — a mid-run /model would
    silently interrupt the agent AND get discarded, a zero-char
    response."""
    return resolve_command(command_name) is not None if command_name else False

The logic is hard: the gateway runner has a safety net that discards any command text that slips into the pending queue. So a /model sent mid-run, if queued, would "silently interrupt the agent AND get discarded," and the user gets an empty response. Conclusion: any resolvable slash command bypasses and dispatches immediately, never queues. This is exactly the danger the cache line guards against: once a control command leaks into conversation history as user text, it could put two user messages back to back, breaking strict role alternation (the invariant from ch.7) and thereby disturbing the cached prefix (ch.6). Bypass routes control commands onto a control channel, keeping them out of history entirely — cutting the danger off at the source.

TUI & desktop

Type hermes --tui and you see a slick terminal UI. It's really two processes: a Node (Ink/React) one paints the screen, a Python (tui_gateway) one runs the agent, and they talk over newline-delimited JSON-RPC on stdio. Better still: the web dashboard's chat page isn't a rewrite — it pipes the same hermes --tui into the browser through a PTY.

🔌 Analogy · a theatre's backstage and stage
Backstage (Python) runs the actors, the script, the cues; the stage (Ink) runs the lights, the set, presenting the play to the audience. The two coordinate via a prompter line (JSON-RPC over stdio): the stage calls "the audience submitted a line" (prompt.submit), backstage feeds the actor's lines back word by word (message.delta). A different venue (the browser)? Don't re-stage the play — just project the whole performance (hermes --tui) onto it.
🌍 Macro · one core, many frontends
TypeScript owns the screen, Python owns sessions/tools/model calls, with a layer of newline-delimited JSON-RPC between them. The same Python backend (the AIAgent core) serves the Ink terminal UI, the PTY embedded in the browser, and the Electron desktop app. Frontends are volatile and iterate fast; the core is stable. JSON-RPC: the one bridge between front and back

On the Python side, a tiny decorator registers a function as a "remotely callable method," then a single dispatcher routes calls:

tui_gateway/server.py959-995 · simplified
def method(name):
    def dec(fn):
        _methods[name] = fn          # register into the global method table
        return fn
    return dec

def handle_request(req):
    rid, m, params = _normalize_request(req)   # validate + unpack
    fn = _methods.get(m)
    if not fn:
        return _err(rid, -32601, f"unknown method: {m}")
    return fn(rid, params)            # dispatch to the registered handler

Any capability the frontend can call just hangs a @method("name") on a function. handle_request receives one JSON-RPC frame, validates it, looks up the method by name, and dispatches. Errors ride the standard JSON-RPC error envelope (-32601 unknown method) — the frontend always gets a structured result or error, never has to parse free text.

Config & Profiles

On one machine you want two fully independent Hermes instances: a "work" one and a "personal" one, each with its own API keys, memory, sessions, skills, gateway — never crossing wires. That's a profile. All its magic boils down to one sentence: before any module is imported, set one environment variable, HERMES_HOME, first.

🔌 Analogy · branches of a hotel chain
One management system (the Hermes code) runs many branches (profiles). Each branch has its own keys, ledgers, rooms (its own HERMES_HOME directory). The moment you walk in, the front desk confirms "which branch are you" (_apply_profile_override sets HERMES_HOME first), and from then on everything you do automatically uses that branch's resources — no charging the wrong ledger, no grabbing the wrong key.
🌍 Macro · one env var decides everything
Switching profiles does one thing: at the module top level of main.py, before any other import, set HERMES_HOME to ~/.hermes/profiles/<name>. After that, the codebase's single path entry point get_hermes_home() automatically points there, and config/keys/memory/sessions/skills are fully isolated. Config itself splits in two layers: behavioral settings go in config.yaml, secrets in .env.

Cron & Kanban

You tell Hermes "every day at 9am, send me today's meeting digest." It must wake on a schedule in the background, run, and deliver — but this must never pollute the conversation you're having: don't stuff the automated task into chat history, don't write it into your "personal memory," and certainly don't let one runaway job freeze the whole scheduler. That's cron's design focus: background automation that breaks neither the cache nor anything else.

🔌 Analogy · the office automation assistant
You hire an assistant (cron) for scheduled chores: a daily clipping, a weekly backup. They work at a separate desk (an isolated session), put results on your desk when done (delivery), but never barge into the meeting you're in (no mixing into the main conversation), and never write these chores into your HR file (skip_memory). If a chore hangs, an inactivity alarm (interrupts after ~10 min of no activity by default) stops them so the next one isn't delayed.
🌍 Macro · scheduling + isolated session + safety valves
Cron is driven by a tick loop (a file lock prevents duplicates), scanning due jobs by the times parse_schedule resolves. Each job spins up an isolated cron session (platform="cron", skip_memory=True), and when done delivers framed results that are not mirrored into the main conversation. Add an inactivity timeout (default 600s) and a half-period catchup window — safe background automation. Kanban is the companion multi-agent work queue, where workers claim tasks via a zero-footprint toolset.

def parse_schedule(schedule):
    """Parse schedule string into structured format."""
    # "30m"/"2h" → once; "every 2h" → recurring; "0 9 * * *" → cron; ISO → at time
    if schedule.lower().startswith("every "):
        return {"kind": "interval", "minutes": parse_duration(...)}
    parts = schedule.split()
    if len(parts) >= 5 and all(re.match(r"^[\d\*\-,/]+$", p) for p in parts[:5]):
        croniter(schedule)                  # validate the 5-field cron expr
        return {"kind": "cron", "expr": schedule}
    if "T" in schedule:                  # ISO timestamp → once
        return {"kind": "once", "run_at": ...}

Four forms cover nearly every need: "30m" (once in 30 min), "every 2h" (every 2 hours), "0 9 * * *" (standard cron, daily 9am), "2026-02-03T14:00" (once at a timestamp). Parsed into once/interval/cron, from which the scheduler computes the next fire.

Safety valves: catchup and the inactivity timeout

Schedulers fear two things: frantically catching up after a miss, and freezing on a runaway. Cron has a valve for each:

cron/jobs.py475-490 · simplified
def _compute_grace_seconds(schedule):
    """Half the schedule period, clamped between 120s and 2 hours."""
    MIN_GRACE, MAX_GRACE = 120, 7200        # 2 hours
    if schedule["kind"] == "interval":
        period = schedule["minutes"] * 60
        return max(MIN_GRACE, min(period // 2, MAX_GRACE))

The catchup window = half the period (clamped 120s to 2h): miss it by a little and it runs once to catch up; miss it by a lot and it fast-forwards instead of piling up catch-up runs (so booting up won't fire dozens of backlogged runs at once). On the other end, a cron session is backstopped by an inactivity timeout: by default, after 600 seconds (10 min) of no activity (no tool call, no stream token) it fires agent.interrupt("Cron job timed out (inactivity)") (tunable via HERMES_CRON_TIMEOUT; an actively-working job can run for hours). So a runaway idle agent can't monopolize the scheduler, while a legitimate long job isn't killed. Plus a .tick.lock file lock means multiple processes won't double-tick.

Eval, batch & trajectory

Hermes isn't just a product — it's Nous Research's research tool: it must batch-run thousands of prompts, persist each conversation's full trajectory (reasoning, tool calls) as training data, and — in a world where models change weekly — write tests that don't break every day. This chapter covers three related things: batching, trajectories, and the testing philosophy.

🔌 Analogy · the factory QC line
Feed raw material (prompts) onto parallel production lines (batch workers); each product records its full process (the trajectory: every reasoning step + every tool call). Before shipping, QC: rejects (e.g. zero reasoning throughout) are discarded, never entering the stock (the training set). And QC checks "does it meet spec" (invariants), not "is it identical to the last batch" (a snapshot) — because the recipe is meant to evolve.
🌍 Macro · a data pipeline + drift-proof tests
Batch: multiple workers run AIAgent in parallel, each prompt independent. Trajectory: turn a conversation into a JSONL training sample carrying conversations/tool_stats/metadata, and filter low quality (drop zero-reasoning samples). Eval: tests lock behavioral invariants, not data snapshots — model catalogs and config versions change daily; lock a snapshot and CI goes red daily.

Plugins, skills & MCP

Across the previous 22 chapters, Hermes's core has barely "grown" — platform adapters, memory backends, model providers, chat skills: the vast majority of capability is bolted on from the edges. This chapter pins down the discipline that runs through the whole book: capability extends at the edges, the core stays a narrow waist. Three mechanisms: plugins, skills, MCP.

🔌 Analogy · the Swiss Army knife vs the toolbox
The core should hold only the few most-used blades (core tools); everything else goes in the toolbox (plugins/skills/MCP), pulled out only when needed. Weld every tool onto the knife and it gets too heavy to carry — because every outing (every API call) must lug the manual for all tools (every tool schema in the context). The more tools, the more the one you actually need drowns in the pile of manuals.
🌍 Macro · the Footprint Ladder
To add a capability, pick the highest rung by smallest footprint: ① extend existing code → ② CLI command + skill → ③ service-gated tool (check_fn) → ④ plugin → ⑤ MCP server (in the catalog) → ⑥ a new core tool (last resort). The further down, the more permanent core surface it occupies. Most capabilities should stop at the top rungs.

Plugins: hook into the core without touching it

A plugin attaches tools/commands/hooks to Hermes through one register(ctx), touching not a single core file:

Security & threat model

Hermes has real privileges: it runs terminal commands, opens browsers, reads/writes files, connects to your messaging platforms. That means the threats are real too — malicious input wants to inject instructions, a poisoned skill wants to contaminate, a single rm -rf / wants to destroy the system. This chapter is a cross-cutting deep-dive: how Hermes uses defense in depth to block these threats layer by layer.

🔌 Analogy · a bank vault's layered security
No single lock is absolute, so a bank layers its defenses: keycard access (command approval), guards who recognize blacklisted faces (dangerous-pattern detection), tellers with tiered permissions (subagent least-privilege), vendor background checks (dependency pinning). One layer falls, the next catches it. The key: don't count on "trusting one smart insider" — because that insider (the model) can be fooled by a single sentence (injection).
🌍 Macro · Defense in Depth
The more capable the agent, the bigger the blast radius, so security is multiple independent layers: ① dangerous commands need approval (hardline patterns aren't even cleared by yolo); ② subagents are stripped of high-risk tools (least privilege); ③ control commands are separated from the data stream at the gateway (anti-injection, ch.18); ④ dependencies are all version-pinned (anti-supply-chain). Never rely on the model to "judge safety" itself — it can be fooled.

    (r":\(\)\s*\{\s*:\s*\|\s*:\s*&\s*\}\s*;\s*:", "fork bomb"),
]

def detect_dangerous_command(command):
    """Returns (is_dangerous, pattern_key, description)."""
    for pattern_re, description in DANGEROUS_PATTERNS_COMPILED:
        if pattern_re.search(command_lower):
            return (True, description, description)
    return (False, None, None)

approval.py is the single source of truth for the dangerous-command system: 12 HARDLINE patterns (delete root, format, raw-disk write, fork bomb…) + 47 DANGEROUS. A match requires user approval. The distinction: /yolo mode can clear DANGEROUS, but HARDLINE red lines always block — leaving a "convenience" door open while welding shut the "will destroy the system" ones. There's also a smart-approval layer: an auxiliary LLM auto-clears low-risk commands to cut the nagging.

Layer two: subagent least-privilege

When delegating to a subagent, a batch of high-risk tools is taken away first:

tools/delegate_tool.py45-53 · verbatim
DELEGATE_BLOCKED_TOOLS = frozenset([
    "delegate_task",  # no recursive delegation
    "clarify",        # no user interaction
    "memory",         # no writes to shared MEMORY.md
    "send_message",   # no cross-platform side effects
    "execute_code",   # children should reason step-by-step, not write scripts
])

Each stripped tool has an explicit reason: no recursive spawning (prevents explosion), no bothering the user, no writing shared memory (prevents pollution), no cross-platform messaging (prevents external side effects). This is least privilege: a subagent gets only the tools sufficient for its task, caging the blast radius at the edge (the security face of ch.13's delegation isolation).

Layer three: supply-chain pinning

The stealthiest attack comes through dependencies. Hermes version-pins every one:

pyproject.toml45-52 · excerpt
# exact pins shrink the blast radius for the next supply-chain attack
dependencies = [
    "openai==2.24.0",
    "httpx[socks]==0.28.1",
    "certifi==2026.5.20",
    "pyyaml==6.0.3",
    # policy: PyPI >=floor,<next_major; Git URL commit SHA; Actions SHA
]

This discipline was set after litellm was poisoned (a supply-chain attack): every PyPI package has an upper bound (>=floor,<next_major), Git deps pin a commit SHA, even GitHub Actions pin a SHA; and core runtime packages (like the openai/httpx above) were further tightened to exact (==) after the Mini Shai-Hulud worm hit mistralai. A compromised upstream package can't sneak in by "auto-upgrading to a malicious new version." Trust nothing that changes automatically — the same paranoia as "don't trust the model to judge safety itself."

🎯 Design trade-off · what this chapter is about
The throughline: defense in depth — the agent has real privileges, each layer independently secure, never relying on the model's own judgment. It mainly treats two inherent LLM constraints:

D·instr=data — to the model, the system prompt, user messages, tool returns, and web content are all tokens with no trusted boundary. A malicious web page/file can masquerade as a "system instruction" and inject. The countermeasure: don't let the model be the judge: control commands are explicitly parsed at the gateway via get_command() and routed on a bypass channel (ch.18); dangerous commands use a regex blacklist, not "ask the model whether this command is dangerous." Move the trust boundary into the code layer.

G·ops — security is an operational responsibility, not a one-off feature: the approval system, least privilege, supply-chain pins, and profile isolation are all continuously maintained infrastructure. Supply-chain poisonings drove repo-wide dependency pinning (litellm set the upper-bound policy; the Mini Shai-Hulud worm later tightened core packages to ==), while the HARDLINE red lines borrow from Mercury Agent's hardened blocklist — the security discipline iterates with real threats.

The anti-pattern: trusting the model to judge safety ("this command looks fine, run it") — the model gets injected, hallucinates, gets lured. Hermes nails security decisions to deterministic code (regexes, frozensets, version numbers), not the model's probabilistic judgment.

📌 Key points
  • Defense in depth: multiple independent layers (approval + least privilege + injection isolation + supply-chain lock); one falls, the next catches.
  • Dangerous-command approval: detect_dangerous_command uses a regex blacklist (12 HARDLINE + 47 DANGEROUS); HARDLINE red lines block even /yolo.
  • Subagent least-privilege: DELEGATE_BLOCKED_TOOLS strips recursive-spawn/interaction/memory-write/cross-platform/script-write, caging the blast radius at the edge (ch.13).
  • Injection isolation (★): control commands explicitly parsed at the gateway via get_command() on a bypass (ch.18), never letting the model judge (D·instr=data).
  • Supply-chain lock: all deps pinned (==/upper bound/commit SHA), trusting nothing that auto-changes (the litellm lesson).

Design principles synthesis

After 24 chapters, you'll see every Hermes design answering the same question: an LLM has 7 inherent flaws (A–G), yet an agent must make the model work autonomously, continuously, and safely in the real world — every piece of engineering treats one of those flaws. This chapter gathers the whole book horizontally: three running design lines + an A–G constraint matrix + a glossary.

🔌 Analogy · connecting stars into constellations
Each earlier chapter is a separate star (system prompt, memory, delegation, gateway…). But they aren't isolated — connect them and clear design lines emerge. See the lines and you graduate from "knowing each part" to "understanding why this machine is shaped this way." That's exactly what an interviewer wants to hear.
🌍 Macro · the one-sentence thesis
A stateless, easily-drowned, instruction-vs-data-blind probabilistic model must work autonomously, long-term, in the real world. All of Hermes's engineering revolves around the 7 hazards (A–G) of that sentence: externalize state, guard the context, push capability to the edges, nail security into code. The three thickest lines: the sacred-cache line, the self-evolution line, the narrow-waist line.

Line 1: per-conversation prompt caching is sacred

The most recurring line in the book. A long conversation reuses the cached prefix each turn; anything that mutates past context voids the cache and doubles the cost. So every part gives way to it:

6the System Prompt is byte-stable for the whole conversation, the cached prefix stays warm
15context compression is the only exception that rebuilds the prefix (so it fires at 50%, with anti-thrashing)
18control commands (/stop//approve) bypass at the gateway, never polluting history or breaking alternation
21Cron spins up an isolated session, not mirrored into the main conversation, so background automation doesn't disturb the cache
23skills inject as a user message (append-only), never into the system prompt

Line 2: self-evolution (cross-session learning)

Hermes's signature — it gets to know you the more you use it. This line upgrades a one-off chat into a continuously growing partner:

9a learning nudge: the model is guided to write reusable solutions into skills
10Curator: a background gardener tracks skill usage, auto-archives stale ones (agent-created only)
11Memory: a MEMORY/USER two-layer + provider, distilling "who you are"
12Cross-session search (FTS5): a new session can dig out relevant slices of old ones

Line 3: the narrow waist (thin core, capability at the edges)

Every core tool is sent on every API call, so each inch of core surface is precious. Hermes makes the core a narrow waist, with capability growing at the edges:

4the narrow-waist philosophy: the core is a thin channel, complexity pushed to both ends
8the tool system + Footprint Ladder: new capability prefers non-core tools
16terminal multi-backend (local/docker/ssh/modal): differences caged in edge subclasses
1728+ platform adapters translate differences into a unified MessageEvent; the core knows only the abstraction
23plugins/skills/MCP: even mainstream platform adapters are edge plugins

🎯 The meta-design philosophy · the book's ledger of trade-offs
Collapse the 7 constraints into one sentence: a stateless, probabilistic, easily-drowned, instruction-vs-data-blind core + all determinism externalized to code and files.

A·lost-in-the-middle B·statelessness C·hallucination D·instr=data E·brittle output F·error accumulation G·ops

Three sentences to remember Hermes's design DNA: ① externalize state — the core has no memory; memory/skills/profile live in external files, the model is a pure function; ② the context is sacred — the cached prefix is byte-stable, with compression the one exception; ③ narrow waist, thick edges — the core is thin enough to hold in your head, capability all grows at the plugin/skill/MCP edge. Security is the cross-cut: never let the probabilistic model be the judge; the trust boundary is nailed to deterministic code.

The anti-pattern (the book's shared enemy): stuffing state into the model, mutating cached context, making every capability a core tool, letting the model judge safety — each crashes straight into one of the A–G constraints.

📌 Glossary · quick reference
  • A–G constraints: the LLM's 7 inherent flaws (lost-in-the-middle / statelessness / hallucination / instr=data / brittle structured output / error accumulation / ops); every design in the book treats one.
  • cached prefix: the byte-stable head of a long conversation reused each turn; mutate it = void it = double the cost. Compression is the only exception (ch.6/15).
  • narrow waist: thin core, capability at the edges; new capability picks the smallest-footprint rung on the Footprint Ladder (ch.4/23).
  • trajectory: a conversation turned into a JSONL training sample (reasoning + tool calls), for RL/eval (ch.22).
  • leaf / orchestrator: subagent roles; a leaf is stripped of high-risk tools like delegate/memory (least privilege, ch.13/24).
  • skip_memory / catchup / check_fn: cron runs no memory to avoid pollution (ch.21) / the half-period window to catch up a missed job (ch.21) / a tool's service gate, absent until its credential is configured (ch.8/23).
