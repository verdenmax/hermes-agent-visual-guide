Hermes 是什么

Hermes 是 Nous Research 做的 自我进化的个人 AI agent。它官网第一句话就把自己定义成 “唯一内建学习闭环的 agent”：在使用中创建技能、 改进技能、提醒自己把知识记下来、搜索自己的过往对话， 并跨会话越来越懂你。同一套 agent 核心，能跑在 CLI、消息网关、TUI 和桌面 App 上。

🔌 生活类比
把普通 agent 想成一个每天失忆的临时工：今天教会的事，明天又得从头教。 Hermes 更像一个会记工作笔记的老员工：每做完一件棘手的活，就把 “这类问题该怎么解”写进技能手册，把“关于你的事实”写进记忆。 下次遇到同类问题，它翻笔记、几步就搞定——越用越顺手。

它到底是什么

一句话：一个把“经验”沉淀成“能力”的 agent。它不替你训练模型，而是在你用它干活的过程中， 把成功的做法变成可复用的程序性知识。这件事之所以重要，是因为大模型本身有个硬伤—— 两次调用之间它什么都不记得。Hermes 的全部“进化”机制，本质都是在对抗这个失忆。

🌍 宏观理解
同一个 AIAgent 核心，被 CLI、网关（Telegram/Discord/Slack…）、TUI、桌面 App 共用。能力主要长在边缘——通过技能、插件、MCP 扩展， 而核心保持很窄。再加一条铁律：每个会话的 prompt 缓存神圣不可侵犯， 几乎所有设计都要绕着它走。

它为什么不绑定你的笔记本

很多 agent 只能跑在你开着的那台电脑上。Hermes 的设计目标之一是把 agent 从单机里解放出来： 它能跑在 5 美元的 VPS、GPU 集群，或 serverless 环境上——闲时休眠到几乎零成本、有请求才唤醒。 于是你可以一边在 Telegram 上发消息，一边让它在云端 VM 上默默干活，关掉手机它也不停。

它还模型无关：Nous Portal、OpenRouter（200+ 模型）、OpenAI、本地端点都行，一条 hermes model 命令切换，不动一行代码。这种“位置自由 + 模型自由”，是它能成为 长期个人 agent 的前提——你的助理不该被锁在某台机器、某个厂商上。这一点也呼应了它的部署哲学： agent 的运行环境本身就是可插拔的，从本机一直到 serverless 沙箱：

进化的发动机：学习 nudge

这套闭环有个关键问题：大模型不会主动反思“我刚学到的东西要不要存起来”。Hermes 的解法很巧—— 在你和它干活到一定轮次时，往对话末尾塞一条反思提醒（nudge）：“你是不是学到了一个可复用的流程？ 要不要用 skill_manage 存成技能？”这条提醒踩着恰当的时机触发、每个会话只发一次，不打扰你。

更妙的是它注入的位置：nudge 是一条追加到消息末尾的普通 user 消息， 不改动 system prompt、不重建上下文。这正是为了守住那条铁律——prompt 缓存不能破。 “提醒 agent 去学习”这件看似简单的事，背后是一个既要驱动进化、又不能破坏缓存的精细权衡（细节见第 9 章）。

③ 关键点

🎯 设计取舍 · 本章围绕什么
全书有三条主线贯穿每个部件：① 自我进化（把经验变能力）、 ② prompt 缓存神圣不可侵犯（长对话每轮复用缓存前缀，绝不中途改写）、 ③ 窄腰架构（核心小、能力在边缘长）。

它对抗的 LLM 固有约束：B·无状态—— 模型两次调用之间零记忆，所以“状态”必须外置到技能、记忆、会话库里。这就是“进化”存在的根本理由。

📌 本课要点
  • Hermes = 自我进化的个人 agent：把经验沉淀成技能（怎么做）和记忆（是什么）。
  • 进化在模型之外：底层模型无状态、固定；靠 nudge + 技能 + 记忆 + 搜索 + curator 协作。
  • 同一 agent 核心驱动多端；能力长在边缘，核心保持窄腰。
  • 一条铁律：每个会话的 prompt 缓存神圣不可侵犯——后面几乎每章都在为它让路。

你在和什么打交道（上）

在拆解 Hermes 的设计之前，先认清我们在和什么打交道。一次大模型调用，本质是一个 无状态、按 token 看世界、凭概率续写的函数：你把上下文递进去，它吐出最可能的下一段文字。 它很强，却带着一组“出厂自带”的硬约束。这一章讲单次调用层面的三类—— 注意力与上下文(A)、无状态与非确定(B)、token 表示层(E)。 它们是后面几乎每个设计的“病因”：你会看到 Hermes 的很多取舍，都是在顺着这些约束设计，而不是硬刚。

🔌 生活类比
把一次调用想成请教一位博学却“每次失忆”的专家：你把所有背景写在一张纸上递进去， 他只读这张纸、给一个回答，然后忘光一切。下次再问，得重新递一张纸。而且他读长文时， 最在意开头和结尾，中间容易扫过去；他还不是一个字一个字读，而是按词块(token)读， 所以“strawberry 里有几个 r”这种字符级问题，他反而容易答错。

A · 注意力与上下文：不是“塞得下”就行

🌍 宏观理解
模型的注意力不是均匀铺在整个上下文上的。位置、长度都会显著影响它“看见”了什么、看得多准。 把上下文当成一种稀缺资源来经营，是 agent 设计的第一课。

① 中间遗失（lost in the middle）。大量实验发现：模型对上下文的开头和结尾 注意力最强，中间最弱。把关键指令埋在长上下文中段，常常“看不见”。对策很直接—— 关键指令在头、尾各放一遍；最相关的检索片段放边缘，别塞在中间。

同一条关键指令，放在不同位置时的“被看见”强度（示意）：
开头·强中间·弱中间·弱中间·弱结尾·强 U 型注意力

② 上下文腐烂（context rot）。就算窗口没满，质量也会随长度下降：指令遵循度变差、幻觉变多。 所以压缩不只是为了省钱/塞得下，更是为了质量——一份精简聚焦的上下文，往往胜过“把什么都塞进去”。 检索也一样：要精度优先于召回，无关文档会“分散注意力”。

③ 长度 → 延迟与成本。上下文越长，每一步越慢、越贵。交互式 agent 必须有上下文预算， 不能无脑堆历史。这条直接催生了 Hermes 的迭代预算（第 5 章）与上下文压缩（第 15 章）。

⚠️ 反直觉
“给它更多上下文”不总是更好。超过某个点，更多上下文 = 更差的回答。少而精 > 多而杂。

B · 无状态与非确定：它没有记忆，也不保证可复现

🌍 宏观理解
两次调用之间，模型什么都不记得；而且同样的输入，未必给同样的输出。 这两点决定了：“agent”本质是围绕一个无状态、非确定函数做的工程编排。

① 无状态。模型不在调用之间保存任何东西。所有“记忆”都必须外置（写进文件/数据库） 或每次重发（塞进 messages）。这正是 Hermes 要有技能、记忆、会话库的根本原因—— 它们是那个失忆核心的“外接硬盘”（第 11、12 章）。

第 1 次调用
读完 messages，回答
->
调用结束
模型忘光
->
第 2 次调用
必须重发全部上下文

② 非确定性。即便温度设为 0，同一输入也可能产出不同结果。所以不能依赖可复现： 要设计重试、校验、工具幂等，用容忍式测试而非逐字断言。

③ 自回归、不可回退。模型逐 token 生成，一旦开了个错头，往往会“承诺并自圆其说”， 把错误一路编下去。对策：先规划、后执行，把“修订”放到单独一遍，并给足推理空间（思考 token）再出答案。 这呼应 Hermes 的委派 / 规划-执行分离（第 13、14 章）。

E · token 表示层：它按词块看世界

🌍 宏观理解
模型不是按字符、更不是按数字理解输入的，而是按token（词块）。 这决定了它在精确字符、计数、算术上天生不可靠——这类活该交给工具。

① 分词导致字符/计数/数学弱。“strawberry 有几个 r”、精确子串匹配、多位数乘法，模型都容易错， 因为它看到的是 str+aw+berry 这样的词块，不是单个字母。 对策：别让模型做精确计算或数字符，交给计算器 / 代码执行（第 8 章工具系统）。

“strawberry” 在模型眼里（示意分词）：
strawberry → 看不清“有几个 r”

② 结构化输出脆弱。模型天生输出散文，不是类型化数据。要可靠拿到 JSON，得用 function calling / JSON mode / 语法约束 + 校验 + 修复回路（第 7 章）。

③ 最大输出截断。长输出可能从结构中间被砍断。对策：分块生成、续写， 按输出预算设计任务，别指望一口气吐出超长结构。

③ 关键心法

🎯 设计取舍 · 本章围绕什么
这一章不讲某个 Hermes 部件，而是立一个认知地基：把模型当成一个有确定物理特性的“零件”来对待。 它无状态、按 token 看世界、注意力两头重中间轻、还不保证可复现。

对应的 LLM 约束： A·中间遗失A·上下文腐烂 B·无状态B·非确定 E·分词。记住它们，后面每个设计你都能看懂“在治哪个病”。

📌 本课要点
  • 注意力不均匀：关键信息放头尾；上下文少而精，长了会腐烂。
  • 无状态 + 非确定：状态必须外置/重发；要重试、校验、工具幂等。
  • 自回归不可回退：先规划后执行，给推理空间，错头难回。
  • 按 token 看世界：精确字符/计数/算术交给工具，结构化输出要校验。
  • 核心心法：顺着模型的物理特性设计，而不是硬刚。

你在和什么打交道（下）

上一章把模型当成一个“零件”看它的物理特性。这一章换个视角：当你把这个零件组装成一个 会自主、多步行动的 agent 时，又有一批问题被放大。它们是 agent 工程里 “最痛”的部分——真实性(C)、指令遵循的怪癖(D)、 多步自主(F)、运维(G)。其中两个标了 ⭐⭐： 指令与数据分不开和误差累积，几乎是所有 agent 事故的源头。

🔌 生活类比
现在让那位“失忆专家”连续做 20 步，每一步都基于上一步的结果。麻烦来了：他编造时也很自信 （你分不出他在猜还是在知道）；他读到纸条里夹带的“指令”会照做（哪怕那是别人塞的）； 他还爱顺着你说（你一质疑他就改口）。20 步下来，小错叠小错，结果可能已经面目全非。

C · 真实性：它会自信地编造

🌍 宏观理解
模型是个推理器，不是事实来源。它会“幻觉”，而且校准很差——“我很确定”和“它真的对” 之间没有可靠关系。对它产出的事实、ID、URL，都要用工具核实，别照单全收。

① 幻觉 + 校准差。模型会编出不存在的 API、引用、数字，且语气一样自信。对策：用检索接地、 要求引用来源、用工具验证。这正是 Hermes 强调“检索接地”(第 12 章)、把事实交给工具的原因。

② 上下文中毒(context poisoning)。一旦一个幻觉进了对话历史，后面会被反复当成事实强化。 对策：别把模型未验证的断言又当事实喂回去；不可信内容要隔离、标注来源。

③ 知识截止。模型只知道训练截止前的世界。对策：用工具取新数据 + 在 system prompt 里 注入当前日期(Hermes 正是这么做的，见第 6 章)。

⚠️ 关键
“我很确定”≠“它是对的”。模型的自信度和正确率不挂钩。不要把它的话当事实，要核实。

D · 指令遵循的怪癖：最重磅的坑

🌍 宏观理解
模型无法可靠区分“指令”和“待处理的数据”——这是提示注入的根源，也是 agent 安全的头号问题。

① ⭐⭐ 指令与数据分不开。工具输出、网页、检索文档、用户文件……任何进入上下文的内容，都可能夹带指令劫持 agent。 对策必须组合拳：最小权限工具、危险动作人在环、隔离/定界不可信内容、 把规划器(特权)与数据处理器(隔离)分离。Hermes 的网关有两道守卫(第 18 章)、委派做权限隔离(第 14 章)。

② ⭐ 谄媚(sycophancy)。模型倾向同意用户，一被反驳就改口。这让“审查/验证”类 agent 极其危险—— 它会顺着你说“你这段有 bug 的代码没问题”。对策：用独立的批评者、对抗式设问，别让用户的断言污染验证(第 14 章 background_review)。

③ 生成-验证差。模型常常“验证”比“生成”强。所以让另一遍/另一个 agent 专门做核查， 而不是自己一口气写完不回头。

④ 提示脆弱。措辞、格式的小改可能让行为大变。对策：把提示当代码——版本化、配 eval 测试集(第 22 章)。

F · 多步自主：agent 最痛的地方

🌍 宏观理解
让模型连续自主跑很多步，错误会累积、目标会漂移。这是“窄而专的 agent 优于放任式长自主”的根本原因。

① ⭐⭐ 误差累积(error compounding)。每步 95% 可靠，20 步连乘只剩 ≈36%。对策：保持回路短、 每步验证、任务分解、设检查点。Hermes 的委派(第 13 章)正是用“短回路 + 分解”对抗它。

每步 95% 可靠，连乘 N 步后的整体成功率：
1 步 95%5 步 77%10 步 60%20 步 36% 误差累积

② ⭐ 长程规划漂移。长任务里 agent 会忘记目标、绕圈、死循环。对策：显式目标追踪(todo)、 周期性“重新接地”(重述目标+进度)、步数预算 + 防循环(第 9 章 todo、第 5 章迭代预算)。

③ 工具越多越不准。工具数量↑、语义重叠↑ → 选择质量↓。对策：小而正交的工具集、清晰描述、 分组/动态加载。这正呼应 Hermes 的窄腰 + Footprint Ladder(第 8 章)。

项目全景地图 · 窄腰

Hermes 很大——CLI、消息网关、20 多个平台、TUI、桌面 App、cron、kanban……但它有一个清晰的形状： 窄腰(narrow waist)。中间是一个很窄的核心（一份共享的核心工具 + 一个 AIAgent）， 两端却很宽——多种前端在上、边缘扩展（技能/插件/MCP）在下。理解了这个形状， 你就理解了 Hermes 大部分架构决策：核心能不长就不长，能力尽量长在边缘。

🔌 生活类比
想想 USB：中间是一个极窄、极稳定的标准接口，两端却接着无数设备—— 键盘、硬盘、手机、打印机。正因为“腰”窄而稳，生态才能在两端疯长。Hermes 也是： 腰（核心工具 + agent loop）保持窄而稳，两端（前端 + 扩展）尽情生长。

一次对话的生命周期

一次对话从你敲下的一句话，到 Hermes 给出最终回复，中间到底发生了什么？答案是一个同步主循环：模型每"想一步"就可能调用工具，工具结果被追加回消息列表，再喂回模型——如此往复，直到模型不再调用工具、直接给出最终回答。这个循环是整个 agent 的心脏，它住在 agent/conversation_loop.py 里，并带着三个硬约束：可中断、有迭代预算、严格的消息角色交替。本章把这颗心脏拆开给你看。

🔌 生活类比
把一次对话想成一条工厂流水线：原料（你的消息）进来，沿传送带走过一个个工位（每个工位 = 模型调用一次工具），工位加工完把半成品放回传送带继续往下走。流水线是同步的——上一个工位没干完，下一个不会启动。它还装了两个安全装置：一个计数器（最多过 N 道工序就强制停，免得空转烧钱），一个急停按钮（你随时能喊停，机器立刻松手）。Hermes 的对话循环，就是这条带安全装置的流水线。

宏观：一个同步的 while 循环

🌍 宏观理解
别被 Hermes 的体量吓到——一次对话的核心是一个普普通通的同步 while 循环。每转一圈做五件事：检查中断 → 消费一格预算 → 调一次模型 → 若有 tool_calls 就执行并把结果追加回 messages → 没有 tool_calls 就收尾。所有"记忆"都装在 messages 这个列表里，每轮整体重发给模型——因为模型本身两次调用之间什么都不记得（第 2 章 B·无状态）。循环不重建上下文、只往末尾追加，这正是为了守住 prompt 缓存（第 6 章）。

主循环：while 条件与循环体

循环"能活多久"由 while 条件决定，"怎么退出"则有三条路。先看条件和循环体（conversation_loop.py:589 与 594-614）：

读懂这十几行，三种退出方式就一目了然：① 达到 max_iterations——api_call_count < max_iterations 不再成立；② 预算耗尽——consume() 返回 False，打上 budget_exhausted 并 break；③ 用户中断——循环顶部发现 _interrupt_requested 为真，打上 interrupted_by_user 并 break。注意中断检查放在每圈最开头，所以哪怕模型正排着一长串工具，也能在下一圈被立刻截停。

诚实标注：grace call 是预留钩子，核心未主动触发

_budget_grace_call 字面上像"预算耗尽还会宽限多跑一轮"，但请别被注释误导。核心代码只有三处碰它：agent_init.py:525 初始化置 False、conversation_loop.py:589 的 while 条件读它、608-609 在循环体里消费后又置 False——没有任何核心代码把它设成 True。换句话说它是一个预留放行钩子、核心未主动触发，正常对话里它永远是 False，不会让循环多跑一轮。预算耗尽后真正给模型"收个尾"的逻辑，由独立的 _handle_max_iterations 负责，与这个标志无关。

迭代预算：防止失控长循环

第 5 步每调一次模型，就先 consume() 一格预算。这把"门闩"由 IterationBudget 把守（agent/iteration_budget.py），线程安全，parent 默认上限 90、每个 subagent 独立默认 50。它的两个核心方法短得能背下来：

agent/iteration_budget.py37-43 · 56-59
def consume(self) -> bool:
    """Try to consume one iteration.  Returns True if allowed."""
    with self._lock:
        if self._used >= self.max_total:
            return False
        self._used += 1
        return True

@property
def remaining(self) -> int:
    with self._lock:
        return max(0, self.max_total - self._used)

为什么要给循环上预算？因为大模型的错误会逐轮累积（第 3 章 F·误差累积）：一旦规划跑偏，它可能没完没了地调工具、绕圈子，把你的钱烧光也到不了终点。有界的迭代数就是一道硬刹车——既封住成本与延迟（A·延迟成本），也给失控的长循环兜了底。一个贴心设计：execute_code 这种"编程式工具调用"会调 refund() 把那一格退回去，所以它不占正常的对话预算。

可中断：随时能喊停

循环每圈开头都查 _interrupt_requested，那么这个标志是谁设的？是 interrupt()（run_agent.py:2400-2440）：它把 _interrupt_requested = True，记下中断消息，并把信号级联给正在执行的工具线程与子 agent，让 in-flight 操作尽快收手。最典型的触发场景在消息网关：当某个会话的 agent 还在跑、你又发来一条新消息，网关就调 running_agent.interrupt(new_message.text)——于是它在下一圈循环顶部被截停，转身处理你的新指令。

每圈往 messages 里追加的，无非两类消息。assistant 消息由 chat_completion_helpers.py:885-890 构造，带着 content、reasoning、finish_reason 三件套；tool 消息由 tool_dispatch_helpers.py:336-343 构造，带着 name、content、tool_call_id。四种角色各司其职：

🎯 设计取舍 · 本章围绕什么
本章围绕对话循环的三条主线：可中断 + 迭代预算 + 严格角色交替不变量。可中断让长任务随时能被新消息接管；迭代预算给"放手让模型自己干"上了硬上限；严格交替既满足 provider 的格式要求，又顺手守住了缓存。

System Prompt 与缓存

这是全书的核心章。Hermes 几乎每一个设计，最后都要回答同一个问题：会不会破坏 prompt 缓存？本章讲清三件事——system prompt 如何分三层组装、为什么它在一次会话里逐字节不变、以及 Anthropic 的缓存断点如何把多轮对话的输入成本砍掉 ~75%。记住一句话就够了：system prompt 每个会话只建一次，唯一的例外是上下文压缩。

🔌 生活类比
把 system prompt 想成一栋楼的地基与承重墙：开工时浇筑一次，整栋楼盖完都靠它，绝不会盖到一半再砸墙重浇——那样楼会塌。再换个角度看缓存：它像一张提前付清的“前缀月卡”——第一轮把这堵长长的前缀全款算一遍、存进上游缓存，之后每一轮只要前缀一个字节都没变，就只补尾款（新增的那几句话），省下的正是反复重算前缀的钱。一旦你中途改了前缀（换工具、重载记忆、重建 prompt），月卡作废，每轮都得重新付全款。

宏观：按稳定度排好序的三层

🌍 宏观理解
system prompt 不是一坨随手拼的字符串，而是按稳定度排好序的三层：stable（身份 / 工具 / 技能 / 环境，几乎不变）、context（调用方消息 + 项目上下文文件，整个会话固定）、volatile（记忆快照 / 用户画像 / 时间戳，最容易变）。三层用 \n\n 拼成一整串，会话开始时构建一次，缓存在 agent._cached_system_prompt 上，之后每一轮原样复用。把最易变的内容压到最后，是为了即便它变了，也只动到整串的尾巴，不殃及前面已被缓存的前缀。

这条纪律为什么值得全书为它让路？因为它直接决定了成本量级。个人 agent 的对话往往是长会话——几十上百轮，那堵长长的前缀被反复重发。缓存命中时，长对话的边际成本几乎只剩“新增的那几句”；缓存一破，每轮都得从头按全价再算一遍。第 1 章那条“prompt 缓存神圣不可侵犯”的铁律，到这一章终于落到了具体的代码与数字上。

三层结构：越稳定越靠前

组装入口是 build_system_prompt_parts()（system_prompt.py:113），它返回一个 {stable, context, volatile} 三键字典。三层的顺序绝非随意——越稳定的越靠前，越易变的越靠后：

stable · 最稳身份 / 工具 / 技能 / 环境
SOUL.md 或 DEFAULT_AGENT_IDENTITY、工具与计算机使用指引、技能清单（第 9 章）、按模型族的操作指引、环境与平台提示。整个会话几乎恒定。
context · 会话内固定system_message + 上下文文件
调用方传入的 system_message，加上 TERMINAL_CWD 下发现的 AGENTS.md / .cursorrules 等项目文件。会话期间不变。
volatile · 最易变记忆 / USER.md / 时间戳
记忆快照、USER.md 用户画像（第 11 章）、外部记忆 provider 块、时间戳 / 会话 / 模型 / provider 行。最可能变，所以压到三层最后。

为什么非要分三层、而不是把所有东西揉成一段？因为缓存是前缀缓存——它从头开始逐字节比对，遇到第一个不同的字节，就从那里往后全部失效。如果把易变的时间戳、记忆混在最前面，那每轮变一下、整段前缀就废了。分层的本质，是把“几乎不变”“会话内不变”“每轮可能变”的三类内容物理隔离，让易变只落在整串的末尾。

这套“一次构建、整会话复用”的纪律，被 system_prompt.py 的模块 docstring 一字钉死：

turns — only context compression triggers a rebuild.  This keeps the
upstream prefix cache warm.

盯住那句 built once per session … only context compression triggers a rebuild——“每会话只建一次，唯一触发重建的是上下文压缩”。这是本章、也是全书最重要的一条不变量。build_system_prompt_parts() 自己的 docstring 把后果说得更直白：Hermes never re-renders parts of this string mid-session — that's the only way to keep upstream prompt caches warm across turns。

缓存策略：system_and_3 打 4 个断点

有了一条逐字节稳定的前缀，剩下的就是告诉上游“从哪里开始可以复用”。Hermes 只用一种布局——system_and_3：在 system prompt 和最后 3 条非 system 消息上各打一个 cache_control 断点，一共 4 个，同一个 TTL（5 分钟或 1 小时）。它把多轮对话的输入 token 成本降低 ~75%：

agent/prompt_caching.py1-79 · 简化
Single layout: system_and_3. 4 cache_control breakpoints — system
prompt + last 3 non-system messages, all at the same TTL (5m or 1h).
Reduces input token costs by ~75% on multi-turn conversations ...

def apply_anthropic_cache_control(api_messages, cache_ttl="5m", native_anthropic=False):
    messages = copy.deepcopy(api_messages)            # 深拷贝，绝不改原列表
    marker = _build_marker(cache_ttl)
    breakpoints_used = 0
    if messages[0].get("role") == "system":
        _apply_cache_marker(messages[0], marker)      # 断点 ①：system 前缀
        breakpoints_used += 1
    remaining = 4 - breakpoints_used
    non_sys = [i for i in range(len(messages))
               if messages[i].get("role") != "system"]
    for idx in non_sys[-remaining:]:               # 断点 ②③④：最后 3 条非 system
        _apply_cache_marker(messages[idx], marker)
    return messages

这段代码有两个细节值得留意。其一，第一行就 copy.deepcopy 整个消息列表——它绝不在原列表上打标记，断点只加在当轮要发出去的副本上，于是被缓存的那份“干净”消息序列始终不被污染。其二，4 个断点共用同一个 TTL（5m 或 1h）：短对话用 5 分钟够便宜，长会话可切到 1 小时让前缀“保温”更久。无论哪种，断点位置永远是“system + 最后 3 条”这一种布局。

为什么是 ~75% 而不是 100%？因为缓存命中并非免费，而是打折：命中的前缀按缓存读取价计费（通常只有原价的一小部分），没命中的新增尾部仍按全价算。一次多轮长对话里，绝大部分 token 都是反复重发的固定前缀——它们每轮都吃折扣价，于是整段对话的输入成本被压到约四分之一。这也解释了反过来的代价：你只要在会话中途动了前缀一个字节，本轮起前缀整体丢失缓存、按全价重算一遍，对话越往后这一下越贵。所以“别碰前缀”不是洁癖，是真金白银。

守住前缀的最后一道闸：注入前扫描

三层里的 context 会把 AGENTS.md / .cursorrules 等文件原样塞进 system prompt。问题是：这些文件可能来自 clone 来的仓库，里面可能藏着 prompt 注入。一旦进了 system prompt，它就成了会话期固定前缀的一部分，用户没有任何机会拦截。所以 Hermes 在注入之前先扫一遍：

agent/prompt_builder.py46-62 · 简化
def _scan_context_content(content: str, filename: str) -> str:
    # context 文件进 system prompt 之前，先过一遍注入检测

        return f"[BLOCKED: {filename} contained potential prompt injection (…). Content not loaded.]"

注释把理由讲得很清楚：the file would otherwise enter the system prompt verbatim and the user has no chance to intervene——“否则该文件会逐字进入 system prompt，而用户没有机会干预”。命中就替换成 [BLOCKED: …] 占位，把威胁挡在前缀之外。这正对应一条 LLM 固有约束：D·指令=数据——模型分不清“指令”和“数据”，所以喂进 prompt 的外部文本必须先过安检。

这道闸还很克制：它用的是 "context" 这档扫描范围——覆盖经典注入、promptware / C2、角色扮演劫持，但不套用更激进的 strict 档规则（SSH 后门、持久化、外泄 URL）。原因写在注释里：context 文件常来自克隆的仓库（安全研究、基建文档），用 strict 档会把正常内容误杀。安检既要挡住注入，又不能把每一份项目文档都拒之门外——这条边界本身就是一处设计权衡。

② 数据流时序

③ 关键点

🎯 设计取舍 · 本章围绕什么
一条主线压倒一切：★ prompt 缓存神圣不可侵犯。system prompt 每会话只建一次， 唯一允许重建的时机是上下文压缩（第 15 章）。由此推出三条反模式，会话中途绝不做： ① 换 toolset（工具 schema 也是前缀的一部分，一换全废）；② reload memory / 重读 USER.md（会改写 volatile 前缀）； ③ rebuild system prompt（哪怕只动一个字节，整段前缀缓存作废，成本翻几倍）。

它同时治两条 LLM 固有约束：A·中间遗失——三层把最关键的身份 / 指令钉在最前、稳定不动； D·指令=数据——context 在注入前先扫描，别让外部文件冒充指令混进前缀。

消息流与 provider 适配

Hermes 同时支持 Anthropic、OpenAI Codex、Gemini、AWS Bedrock、以及二十多家 OpenAI 兼容后端。可它的核心对话循环从头到尾只认识一种消息格式——OpenAI 风格的 messages。本章讲清这套「统一抽象」是怎么做到的：一张 transport 注册表按 api_mode 分派，四个 adapter 把统一格式翻译成各家方言，reasoning 跨轮原样回传，外加一条贯穿始终的不变量——严格角色交替。

🔌 类比 · 同声传译 + 万能转接头
把核心循环想成一位只说一种语言的指挥官。面对来自世界各地的将领（provider），他不学二十几门外语，而是雇了一组同声传译（adapter）：指挥官永远用「普通话」（OpenAI messages）下令，传译当场翻成对方的方言（Anthropic blocks / Codex responses / Gemini parts / Bedrock converse）；对方的回话再被翻回普通话。转接头（transport 注册表）则负责「这位将领该派哪位传译」——查 api_mode，取对应 transport。加一个新国家，只要再雇一位传译、登记一个转接头，指挥官一个字都不用改。
🌍 宏观 · 一套 messages，多家后端
核心循环（第 5 章）始终在一份 OpenAI 风格的 messages 上工作：{"role": "system/user/assistant/tool", ...}。所有 provider 差异——鉴权、字段名、reasoning 形态、tool schema 方言——都被挡在 transport + adapter 这一层，绝不渗进核心。出站时 build_api_kwargs 按 api_mode 选一个 transport 把统一 messages 翻成各家请求；入站时 normalize_response 再把各家响应归一回同一种 assistant 消息（含 reasoning）。这正是第 4 章「窄腰」哲学的一个具体落点：核心薄、边缘厚。

分派的起点：transport 注册表

每一种 api_mode 对应一个 transport 类。注册表用「懒发现」装载——第一次取的时候才扫描并导入所有 transport 模块，之后命中缓存：

agent/transports/__init__.py21-46 · 节选
def register_transport(api_mode, transport_cls):
    # 四个 transport 模块在文件尾各自调用它登记自己
    _REGISTRY[api_mode] = transport_cls

def get_transport(api_mode):
    global _discovered
    if not _discovered:
        _discover_transports()              # 懒发现：首次才导入所有 transport
    cls = _REGISTRY.get(api_mode)
    if cls is None:
        _discover_transports()              # miss 也重扫，容忍乱序导入
        cls = _REGISTRY.get(api_mode)
    if cls is None:
        return None                        # 允许调用方回退到旧路径
    return cls()

四个 transport——anthropic_messages / codex_responses / chat_completions / bedrock_converse——在各自模块尾部调用 register_transport(...) 把自己挂进 _REGISTRY。注意 get_transport 在 miss 时会再扫一次：因为测试或乱序导入可能让注册表只装了一半，「未命中就重新发现」保证合法 api_mode 永远取得到。取不到才返回 None，让调用方优雅回退——这是为渐进迁移留的口子。

这里有一个值得注意的不对称：transport 有四种（含 bedrock），但 Gemini 不在其中。因为 Gemini 走的是客户端层替换——当 base_url 指向原生 Gemini 时，GeminiNativeClient 直接顶替底层 OpenAI client（agent_runtime_helpers.py），而它的 api_mode 仍是 chat_completions。所以严格说，翻译层是「四个 transport + 一次客户端替换」，adapter 名单（anthropic / codex_responses / gemini_native / bedrock）与 transport 名单并不完全重合——但这一切对核心循环依旧透明。

真正把统一 messages 变成「某一家的请求参数」的，是 build_api_kwargs。它是一组按 api_mode 的硬分支，每一支取对应 transport、调它的 build_kwargs：

出站到入站：一条完整的数据流

1统一 messages（OpenAI 风格，核心循环唯一认得的格式）
2repair_message_sequence_with_cursor —— 每次 API 调用前修复严格角色交替
3build_api_kwargs 按 api_mode 取 transport
4adapter 翻译成各家方言（Anthropic blocks / Codex input / Gemini parts / Bedrock converse）
5provider API 返回响应
6normalize_response 归一回统一 assistant 消息（含 reasoning 三字段）
7append 进 messages —— 回到核心循环，进入下一轮

第 2 步的角色交替修复是整条链的隐形守门员。Provider 普遍要求消息严格交替（不能两条同 role 连续、不能有孤儿 tool 消息），违反就返回空响应、触发重试。repair_message_sequence_with_cursor 在每次 API 调用前跑一遍：删掉没有配对的孤儿 tool 消息、合并连续的 user 消息。它在发送前直接就地修整 live messages（messages[:] = merged，持久化 / SessionDB 都会看到修整结果），只做最小外科式整理、绝不激进重建上下文——避免无谓改写历史击穿缓存（第 6 章）。

① 组件清单（★本章核心，其余跨章节配合）

🎯 设计取舍 · 本章围绕什么
主线：统一抽象 + 严格角色交替不变量——一套 messages，多家后端；并在每次 API 调用前修复交替。它治三条 LLM 固有约束：

E·结构化输出脆弱——function calling / tool schema 统一成 OpenAI 格式，由 adapter 各自翻成方言（xAI 还要现场剥掉它不认的 schema 关键字），把「结构化输出对方言敏感」这件脆弱事关进边缘层； G·推理token不持久——reasoning_details 原样跨轮 replay，让模型「接着上轮想」； D·提示脆弱——统一格式把 provider 差异隔离，核心 prompt 不必为每家后端写一套。反模式：把某家 provider 的特殊字段、特殊鉴权泄漏进核心循环——那等于让指挥官去学方言，每加一家就要改一次核心。

  • transport 注册表：register/get_transport 懒发现，按 api_mode 分派；加后端只需登记 transport + 写 adapter。
  • build_api_kwargs：按 api_mode 四分支出站构造；Bedrock 直连 boto3、Codex 走 /responses，核心循环不感知。
  • reasoning 三处存储：reasoning 内部留存、reasoning_content thinking 回传（缺失 pad 空格防 400）、reasoning_details 原样跨轮透传（约束 G）。

工具系统：注册·门控·分派·预算

工具是 agent「动手」的能力——读文件、跑终端、搜网页、调浏览器。本章讲清一个工具如何从一个 Python 文件变成模型能调用的能力：注册表自动发现、check_fn 服务门控、按 toolset 过滤组装 schema、统一 JSON 返回 + 错误净化、以及 execute_code 退还预算。还要回答一个贯穿全书的问题——为什么每加一个核心工具都要如此谨慎：因为每个工具的 schema，每一次 API 调用都要发送一遍。

🔌 类比 · 机场安检 + 登机口
把工具集想成一座机场。每个航班（工具）要开放登机，得先过两道关：安检（check_fn）——没配齐前置条件（API key、playwright、docker）的航班根本不出现在航班表上；登机口分配（toolset 过滤）——只有这个平台启用的 toolset 里的工具才会摆上台面。乘客（模型）看到的航班表（tool schema），是每次都要重新打印发给塔台的——所以航班越多，每次通讯越贵。这就是为什么「再开一条核心航线」要慎之又慎。
🌍 宏观 · 能力在边缘，核心窄腰
Hermes 的核心工具集刻意保持很窄（第 4 章的「窄腰」）。原因很硬核：每个工具的 JSON schema 都会随每一次 API 调用一起发出去——工具越多，固定开销越高，还会挤占模型注意力。所以新能力的优先级是「Footprint Ladder」：扩展现有 > CLI 命令 + 技能 > check_fn 门控工具 > 插件 > MCP server > 最后才是新核心工具。注册一个工具只要两件事：写 tools/your_tool.py 调 registry.register(name, toolset, schema, handler, check_fn=...)，再把工具名放进某个 toolset——但「该不该放进核心」永远是最贵的决定。

门控：check_fn 让工具「条件不满足就不出现」

组装发给模型的 schema 列表时，注册表会对每个工具跑一遍它声明的 check_fn。返回 False 的工具直接被跳过、不进 schema——条件不满足时它的足迹是零：

这里有两层巧思。其一，check_fn 的结果用 _check_fn_cached 缓存约 30 秒——因为像「探测 docker / 探测 playwright / 探测 API key」这种检查有成本，30 秒 TTL 让你 hermes tools enable foo 后近实时生效，又不必每次调用都重新探测。其二，门控让工具集对一个会话保持稳定：Home Assistant 的工具只在配了 token 时出现，否则连 schema 都不占——这正服务第 6 章的缓存铁律（工具 schema 也是前缀的一部分，会话中途绝不换 toolset）。更上一层，get_tool_definitions（model_tools.py:276）先按 enabled/disabled toolsets 过滤，且结果按 config.yaml 的 mtime+size 记忆化——配置不变就不重算。

分派：统一 JSON 返回 + 错误净化

模型回来一个 tool_call，dispatch 按名字找到 handler 执行。所有 handler 都返回 JSON 字符串，而所有异常都被统一包成 {"error": ...}——而且要先净化：

tools/registry.py390-416 · 简化
def dispatch(self, name, args, **kwargs):
    entry = self.get_entry(name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {name}"})
    try:
        return entry.handler(args, **kwargs)         # handler 必返回 JSON string
    except Exception as e:
        raw = f"Tool execution failed: {type(e).__name__}: {e}"
        # 经 _sanitize_tool_error 净化：异常串里的框架 token / CDATA /
        # 围栏不能作为结构噪声到达模型
        sanitized = _sanitize_tool_error(raw)
        return json.dumps({"error": sanitized})

这段的关键不是「捕获异常」，而是净化。工具的输出——无论成功结果还是报错串——对模型来说都是数据，不是指令。可异常消息里可能偶然带上模型框架的特殊 token、CDATA、代码围栏；若原样塞回去，模型可能把它们当成结构信号误解析。_sanitize_tool_error 把这些剥掉，让报错老老实实当数据。这正对应 D·指令=数据——模型分不清「指令」和「数据」，所以喂回去的工具输出必须先消毒。

第 5 章讲过迭代预算（parent 90 / subagent 50），每轮工具调用消耗一格。但有一个例外——execute_code（程序化工具调用）是 RPC 式的廉价调用，不该吃预算。所以本轮若且仅若所有 tool_call 都是 execute_code，就退还一格：

agent/conversation_loop.py4117-4122 · 节选

学习引擎与技能

Hermes 最招牌的特性是自我进化：它会跨会话变得更懂你、更会做你这一类任务。本章讲清这套学习闭环的入口——每隔约 10 轮（默认值，可配置），在一轮对话结束、响应已经交给你之后，Hermes 会 fork 一个后台 review agent，让它重放刚才的对话，自问「这轮有没有该存下来的技能或记忆？」。最关键的一句话：这一切发生在后台，主对话和 prompt 缓存全程一个字节都不动。

🔌 类比 · 下班后复盘 / 学徒记笔记
把它想成一位敬业的学徒。白天干活时（主对话），他全神贯注把活干完，绝不停下来写日记——那会打断手上的任务。但下班之后（响应已交付），他会翻看今天的工单，把「这一类活该怎么干」沉淀成手册（技能），把「这位客户是谁、喜欢什么」记进备忘（记忆）。第二天上工，他带着更新过的手册，但今天的工作流程一点没被打断过。学习发生在「别的时间、别的人手里」——这正是不破坏缓存的关键。
🌍 宏观 · 程序性记忆 vs 声明性记忆
Hermes 把「学到的东西」分成两类：技能是程序性的——「怎么做这一类任务」（SKILL.md + 脚本/模板/参考），可执行、可复用、按需加载；记忆是声明性的——「用户是谁、当前状态如何」（第 11 章）。两者由同一个后台 review fork 一起产出。技能下沉到边缘（第 4 章窄腰：能力不进核心、按需加载），既不挤占每次调用的工具 schema，也不撑爆上下文。

技能与记忆的分界：写进哪里

到底什么该进技能、什么该进记忆？后台 review 的 prompt 把这条线划得很清楚——这是真实喂给 review fork 的指令原文：

agent/background_review.py233-237 · 节选（review prompt 原文）
Memory captures 'who the user is and what the current situation
and state of your operations are'; skills capture 'how to do
this class of task for this user'. When they complain about how
you handled a task, the skill that governs that task needs to
carry the lesson.

一句话：记忆 = 是谁 / 是什么状态（紧凑、每轮注入），技能 = 怎么做这一类任务（按需加载的可执行手册）。当用户抱怨你某次任务做得不好，正确的反应不是「记一条 memory」，而是去更新管这类任务的那个技能——让教训沉淀进「下次怎么做」。

学习由一个计数器驱动：每轮累加，到阈值就置一个布尔。但关键在于——它不往主对话注入任何文字，只是个标记，等响应发出之后才被消费：

不破缓存的关键：fork 一个独立 agent

消费这个布尔的，是一个独立 fork 出来的 review agent。它的模块 docstring 把不变量钉死：

agent/background_review.py3-8 · 节选

to fire off a daemon thread that replays the conversation snapshot in a
forked AIAgent and asks itself "should any skill/memory be saved
or updated?".  Writes go straight to the memory + skill stores.
Main conversation and prompt cache are never touched.

这就是「自我进化」与「缓存神圣」能并存的全部秘密：review 跑在一个 daemon 线程 + fork 的 AIAgent 里，重放对话快照、自问该不该存，写入直接落进技能/记忆库。主对话的消息序列、system prompt、上游缓存——全程零改动。写发生在别处，新技能要到下个会话才进 stable 前缀。

第 9 章里，后台 review 会不断创建新技能。问题随之而来：技能库会不会越长越乱、塞满一堆只用过一次的「一次性」技能？这就是 Curator（技能园丁）要解决的——它在后台定期巡查 agent 创建的技能，把长期不用的自动降级、归档。但它有一条铁律：永远只归档、绝不删除，而且整个过程不碰主对话的 prompt 缓存。本章也是「数据安全 + 辅助模型隔离」的范本。

🔌 类比 · 图书馆管理员 + 可恢复的暂存库
把技能库想成一座图书馆。管理员（curator）每隔一阵子巡一次馆，把很久没人借的书先挪到「暂存区」（stale），再久没动就搬进地下库房（archived）。但他从不烧书——库房里的书随时能取回。读者钉住（pinned）的常用书永远留在书架上、不受巡查影响。最关键的是，管理员在另一间办公室干活（forked agent），从不打断正在阅览区看书的你（主对话）。
🌍 宏观 · 自我维护，但永不破坏
Curator 的设计被一句话概括：自我维护，但永不破坏。它只治理「后台 review fork 自动创建」的技能（标记 created_by=agent）——你手工建的（不会被标记）、hub 安装的、受保护内置都不碰；它永不删除，最激进的动作是归档（可恢复）；pinned 技能豁免一切自动转换；而且它用辅助模型跑在独立 session 里，从不触碰主对话的 prompt 缓存。这四条不变量，正是「让 agent 自由进化」与「数据绝不丢、缓存绝不破」之间的安全边界。

四条不变量：写在 docstring 里的安全边界

这些约束不是口头约定，而是钉死在模块 docstring 里的严格不变量：

agent/curator.py15-19 · 节选
Strict invariants:
  - Only touches agent-created skills (see tools/skill_usage.is_agent_created)
  - Never auto-deletes — only archives. Archive is recoverable.
  - Pinned skills bypass all auto-transitions
  - Uses the auxiliary client; never touches the main session's prompt cache

逐条拆开：①这里的「agent 创建」特指后台 review fork 自动创建的技能（产权门控见第 9 章，created_by=agent）——你手工建的不会被标记，所以永远安全；hub 安装的无条件豁免，bundled 内置默认（prune_builtins）也只可能被归档、绝不删改；②永不删，只归档，且归档可恢复（搬进 .archive/，随时 restore）；③pinned 豁免一切自动转换；④用辅助客户端、绝不碰主 session 的 prompt 缓存。最后这条把 curator 和「缓存神圣」（第 6 章）直接绑定。

确定性状态机：active → stale → archived

降级的核心是一个纯确定性的状态机——不调用任何 LLM，只看技能「最近一次真实活动」的时间戳：

agent/curator.py276-331 · 简化
for row in skill_usage.agent_created_report():
    if row.get("pinned"):
        continue                                  # pinned 豁免一切转换
    # 活跃锚点：最近活动 -> 创建时间 -> 现在（新技能不会立刻自我归档）
    anchor = last_activity or created_at or now

    if anchor <= archive_cutoff and current != ARCHIVED:
        archive_skill(name)                       # 90 天未动 -> 归档（可恢复）
    elif anchor <= stale_cutoff and current == ACTIVE:
        set_state(name, STALE)                    # 30 天未动 -> 标记 stale
    elif anchor > stale_cutoff and current == STALE:
        set_state(name, ACTIVE)                   # 又被用了 -> 复活

三条转换都基于 anchor（最近活动时间）：30 天没动 active → stale，90 天没动 stale → archived，一旦又被用到则 stale → active 复活。注意 anchor 的回退链 last_activity → created_at → now——保证刚建的新技能不会立刻把自己归档。整段没有一次 LLM 调用，纯靠遥测时间戳（第 9 章的 skill_usage 喂数据），既便宜又可预测。

辅助模型隔离：fork 一个独立 agent

除了确定性降级，curator 还能（可选）跑一次 LLM「合并」pass——把零散技能并成「伞形」大技能。这一步用一个完全独立 fork 的 AIAgent，跑在辅助模型上：

agent/curator.py1826-1845 · 简化
review_agent = AIAgent(
    model=_model_name, provider=_resolved_provider, ...,
    max_iterations=9999,           # umbrella 扫描可能 50-100 次调用
    quiet_mode=True,
    platform="curator",           # 独立 platform，独立 prompt cache
    skip_context_files=True,
    skip_memory=True,
)
review_agent._memory_nudge_interval = 0    # 禁止递归 nudge
review_agent._skill_nudge_interval = 0     # curator 绝不能再 spawn review

注意几处关键设定：platform="curator" 让它有独立的 prompt 缓存，不与主对话共享；skip_context_files / skip_memory 让它不加载主 session 的上下文与记忆；两个 nudge_interval = 0 禁止递归——curator 绝不能再触发自己的 review（否则无限套娃）。它的 stdout/stderr 还被重定向到 /dev/null，连终端噪声都不漏。更重要的是：这次 LLM 合并 pass 默认是关闭的（opt-in）——因为它每跑一次都烧辅助模型 token，确定性降级才是常态、始终在跑。

🧩 协作机制 · 各组分如何咬合实现「自我维护而不破坏」
① 组件清单（★本章核心，其余跨章节配合）
本章核心：maybe_run_curator（inactivity 门控触发）、apply_automatic_transitions（确定性状态机）、forked review AIAgent（platform=curator 的辅助模型隔离）、archive/restore（可恢复归档）。跨章节配合：curator 只园丁 background review fork 创建的 agent-skill（第 9 章产权门控）；forked agent 用独立 prompt cache、不碰主对话（第 6 章缓存 + 与第 9 章 fork 同源的「学习在别处」机制）；降级靠 skill_usage 遥测时间戳（第 9 章的 use/view/patch 计数喂回这里）；辅助模型走独立 aux 槽、与主 runtime 隔离。
② 数据流时序
agent 空闲 + 距上次运行超过 interval_hours → maybe_run_curator → apply_automatic_transitions（读 skill_usage 时间戳，确定性降级，pinned 跳过）→ 持久化 .curator_state →（仅当 opt-in consolidate 开）fork 辅助模型 review agent 做 umbrella 合并 → 全程主对话 / 缓存零改动，最坏结果只是把技能搬进可恢复的 .archive/。
③ 关键点
「自我进化」需要「自我维护」兜底，否则技能库会无限膨胀（误差累积）。但维护本身必须零破坏：永不删（只归档）、不碰主缓存（fork 独立 session）、pinned 豁免、LLM 合并 opt-in 省成本——每一条都在「主动维护」与「绝不伤害用户资产/性能」之间划线。
🎯 设计取舍 · 本章围绕什么
主线：自我维护但永不破坏（数据安全 + 辅助模型隔离）。它治两条 LLM 固有约束：

F·误差累积——不加约束的自动学习会让技能库无限膨胀、互相污染；curator 用确定性状态机持续修剪，把熵控制住； G·运维——它是一套后台运维系统：inactivity 触发、状态持久化、可恢复归档、备份/回滚。两者都服务「缓存神圣」（第 6 章）：维护跑在 forked 辅助 agent 里、独立 prompt cache。反模式：让 curator 删除技能、或在主 session 里跑维护——前者毁掉用户资产，后者击穿缓存。

📌 本课要点
  • 四条不变量：只碰 agent 创建 / 永不删只归档（可恢复）/ pinned 豁免 / 不碰主缓存。
  • inactivity 触发：非 cron daemon——agent 空闲 + 距上次超过 interval_hours 才由 maybe_run_curator 跑。
  • 确定性状态机：active→stale→archived（30/90 天）+ 复活，纯看活动时间戳、零 LLM。
  • 辅助模型隔离：forked AIAgent platform="curator"、独立 cache、skip_context/memory、nudge=0 防递归。
  • LLM 合并 opt-in：烧 token 的 umbrella 合并默认关；确定性降级始终在跑。
